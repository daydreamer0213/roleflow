# 简历驱动工作流

## 目标

让不同求职者都从自己的简历开始，而不是从固定关键词开始。系统只读取岗位信息，不自动点击立即沟通、不自动发送招呼语、不自动投递。

## 用户路径

```text
上传简历
  -> 本地提取文字（TXT / MD / DOCX / PDF）
  -> LLM 或离线 mock 生成候选人画像
  -> LLM 或离线 mock 推荐 Search Plan
  -> 用户编辑并确认筛选条件
  -> 按确认的 Plan 读取 BOSS 岗位
  -> 规则硬筛 + JD 匹配 + 生成岗位操作台
  -> 用户标记已投 / 跳过 / 无回复 / 跟进
```

## 本地数据边界

- 上传的原始文件只在解析期间写入 `.runtime/resume-parse`，随后删除。
- 提取后的简历文本、候选人画像、Search Plan、岗位和投递状态保存在 `data/jobs.sqlite`。
- API Key 只能从环境变量读取，不能写入配置或 SQLite。
- 没有真实模型 Key 时，`mock` 会给出可编辑的离线建议，并在页面上显示来源；不能把它当成语义分析结果。

## Search Plan 的作用

每个 Plan 包含城市和 BOSS 城市编码、薪资范围、经验可冲策略、活跃度、搜索关键词、软排除词、硬排除词及扫描规模。

- 软排除词：降低排序并显示风险，例如“销售”“培训”。
- 硬排除词：直接压到低价值，例如“培训贷”。
- 关键词、目标方向和简历技能会动态构成正向评分，不再把 AI/RAG 关键词强行套到所有求职者身上。

## 验收边界

离线自动验收覆盖上传表单、画像、Search Plan 保存、按 Plan 样例扫描、岗位页与状态标记。真实 BOSS 扫描仍需用户先登录专用 Edge，并在实际页面上检查城市、BOSS 活跃度、重复出现和岗位链接。
