# 简历驱动首次使用流程

## 目标

新用户从自己的简历和求职偏好出发，不从内置 AI 关键词出发。系统负责给初始建议，用户只需要确认影响搜索的少量条件。

## 用户路径

```text
打开工作台
  -> 配置并实测模型连接
  -> 上传简历或粘贴文本
  -> 本地提取和质量诊断
  -> 模型生成 CandidateProfile
  -> 先保存画像与简历版本
  -> 模型推荐 Search Plan
  -> 用户选择城市并调整经验、薪资、关键词
  -> 保存方案
  -> 启动 BOSS 只读扫描
```

模型配置位于使用链路前面，但不会封锁历史数据：未配置模型时仍能查看已有岗位、投递记录和诊断信息。

## 简历入口

- TXT / MD：直接读取 UTF-8 文本。
- DOCX：读取正文、页眉和页脚。
- PDF：读取文本层；疑似扫描件时明确提示粘贴文本，不伪装成已完成 OCR。
- 文本粘贴：最稳定的兜底，并提供包含求职意向、教育、经历、项目、技能和优势的模板。

原文件只在解析时短暂写入 `.runtime\resume-parse`，随后删除。提取文本和结构化结果存入本地 SQLite。

## 画像原则

- 用户输入视为已确认事实，不要求逐字段重复确认。
- 模型只整理材料，不得编造公司、时间、职责、学历、结果或技能。
- 页面不在解析阶段点评简历，也不追问毕业月份、四级分数、团队人数等与岗位筛选无关的信息。
- GAP、离职原因、到岗时间和短期项目口径不能仅凭简历生成；只有 HR 实际提问时才按需询问一条事实。

## Search Plan

模型可以推荐：

- 目标岗位方向和 A/B/C 搜索关键词。
- 薪资范围、经验范围和 3–5 年可冲策略。
- 全职类型、学历、排除词和扫描规模。

城市由用户选择；只有简历明确写出求职城市时才能预填。保存后的方案绑定具体画像版本，画像更新后旧方案标记为 stale，必须重新确认，避免用旧条件扫描。

## 验收

离线回归覆盖文件解析、粘贴文本、模型连接、画像持久化、简历版本、方案保存、过期依赖、队列和投递状态。真实 BOSS 只在离线检查通过并冷却充分后做 3–5 条串行只读验收。
