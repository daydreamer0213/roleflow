# RoleFlow 全链路整改计划

更新时间：2026-07-15

## 目标

依据 `docs/runtime_flow_review.md` 的 P-001 至 P-050，修复来源采集、数据可见性、画像与搜索计划、LLM 匹配、最终排序、按需沟通和工程诊断，并完成真实 BOSS 小批只读验收。

完成不等于“代码写完”或“旧测试通过”。只有自动化回归、数据库一致性检查和真实页面抽样同时通过，目标才算完成。

## 不变边界

- 不自动点击沟通，不自动发送消息，不自动投递。
- BOSS 操作保持一个已登录 Edge 标签页串行执行；滚动/卡片约 0.9-2 秒，导航约 1.8-3.2 秒，关键词切换约 2.5-4.5 秒，每 18-26 次外部动作随机冷却 4-7 秒。
- 不清空用户主数据库，不覆盖现有已投、跳过、约面和反馈历史。
- 不把双休、外包、2-3 年或低薪 3-5 年可冲简单改成全局硬排除。
- 不让用户反馈自动学习并改变排序；反馈只进入诊断样本。
- 不要求用户在首次使用时逐字段重复确认简历事实。

## 阶段 A：来源事实与浏览器采集

问题范围：P-004、P-022、P-023、P-024、P-026、P-027、P-028、P-030、P-034。

实施内容：

1. 扫描前返回 bridge、扩展、候选标签页、登录态、页面类型和 DOM 结构；一次运行锁定同一个 tab ID。
2. Search Plan 城市只能保存可解析的平台城市；求职类型映射 BOSS `jobType`，薪资/经验目录变化后按语义范围重新映射，不静默取消原生筛选。
3. 列表滚动动态寻找真实滚动祖先，优先 `document.scrollingElement`；以卡片数量/DOM 增长和页底状态决定继续或结束。
4. 左卡只采集标题、公司、地点、薪资、经验、学历和链接；点击左卡后等待右栏岗位 ID/标题切换，再读取完整 JD、技能标签和招聘者活跃状态。只有必要字段缺失时才滚动右侧 `.job-detail-container` 补读。
5. 除无效链接、明确异地、实习/纯算法等确定性硬排除外，所有跨关键词去重后的唯一岗位都需要右栏详情。空的 `.boss-online-icon` 只可作为在线信号，不能替代周/月等文字状态。
6. 按“城市 + 关键词 + 薪资 lane”建立检查点；单目标失败记录后继续，已采集结果立即落库。
7. 保存岗位级 refresh attempt、错误码、次数和冷却；活跃度与 observation 时间绑定。只有采集时有效、后来因快照老化超过 3 天的岗位进入活跃探针，采集时已是周/月/半年内活跃的岗位直接留在不建议历史池。

验收标准：

- 自动化测试覆盖：真实滚动祖先、卡片追加、双栏切换事件、右栏长 JD 滚动、活动图标、标签页锁定、单目标失败继续和检查点恢复。
- 模拟超过 15 张卡片时能继续加载，不以固定三轮“数量没变”过早结束。
- 真实验收每次只抽样 3-5 张卡片，页面可见的标题、公司、地点、薪资、经验、活跃状态和链接与落库结果逐项一致；页面未提供才允许 unknown。其余覆盖使用保存的 DOM 夹具和假浏览器回归，避免为测试反复访问账号。
- 右栏切换超时最多一次退避重试，失败保留左卡事实并可定位，不重复机械点击。
- 删除旧的 A/B/C 4/3/1 低详情配额；日常扫描改为 A/B 每词 40/30、全局 220，达到安全上限的岗位保留为待读详情，不进入语义推荐。
- 真实 BOSS 小批验收使用单标签串行、随机间隔和分段冷却；出现验证页、登录异常或页面丢失立即停止，不切换标签绕过，全程没有自动沟通或投递。

## 阶段 B：唯一岗位、快照与队列可见性

问题范围：P-029、P-031、P-033、P-042、P-043、P-044、P-049。

实施内容：

1. SQL 先按 Plan + job ID 选最新 observation，再对唯一岗位分页；取消 observation 前置 `LIMIT 500`。
2. 来源内容哈希只包含平台事实和清洗 JD；模型、画像、规则和分数使用独立 analysis revision。
3. 经验优先使用平台结构化字段；缺失时只从任职要求区块提取，不让正文任意数字覆盖卡片经验。
4. 紧凑队列增加分页和总数，区分本轮新增、本轮重复和历史未处理，并展示首次/最近出现时间。
5. “保留”改成明确的稍后日期，紧凑入口默认 7 天后；到期前不重新进入当前队列。
6. 删除 `renderQueuePage` 中 return 后不可达的旧实现。

验收标准：

- 构造超过 600 条 observation、超过 100 个唯一岗位的数据，所有未处理岗位均可分页访问，重复记录不会挤掉唯一岗位。
- 每个分组显示“当前范围 / 总数”，超过 30 条可以继续访问。
- 新扫描、重复出现、历史积压和详情刷新批次在页面上可区分。
- 修改模型或规则不再触发 `detail_changed`；只有平台事实/JD 改变才触发。
- 卡片经验 1-3 年且正文出现非门槛“5 年”时仍保持 1-3 年。
- 点击“7 天后再看”后岗位立即退出当前队列，到期后重新出现。

## 阶段 C：首次使用、画像、简历版本与 Search Plan

问题范围：P-001、P-005、P-006、P-008、P-009、P-010、P-011、P-012、P-013、P-014、P-015、P-016、P-017、P-018、P-019、P-020、P-021、P-025。

实施内容：

1. 模型设置增加真实连通性检测；厂商 Key 独立保存，DPAPI 文件必须实际可解密；公开设置与 Key 采用可恢复保存流程。
2. 首次页面只显示厂商、模型和 Key；高级 base URL、超时、mock 放到高级设置。
3. 模型预设使用可维护推荐值，允许自定义模型；预设变更不覆盖用户选择。
4. 页面明确说明完整简历文本会发送给所选模型厂商；模型不可用时仍允许进入只读历史和规则降级，不把扫描入口全局锁死。
5. 简历本地提取增加轻量质量诊断；不再宣称未实现 OCR，扫描 PDF 直接推荐粘贴文本兜底。
6. CandidateProfile 保留教育、工作/实习、项目贡献边界、技能和个人优势等匹配事实；不生成 GAP、离职或短期项目原因。
7. 简历版本保存真实正文/文件引用并参与匹配，新增版本要重新分析；画像版本和 Search Plan 记录依赖关系。
8. 画像修改后只提示方案已基于旧画像，不静默覆盖人工方案；重生成由用户触发。
9. 筛选页面改为同一次“保存并扫描”，或在表单脏时禁止启动旧方案。
10. 没有明确求职地点时不默认广州；当前用户因材料明确写广州优先可预填广州。

验收标准：

- 假 Key、错误模型、错误 base URL、无余额/429、超时分别显示可理解错误，不能被标为已可用。
- 切换厂商不会复用旧厂商 Key；损坏 DPAPI 文件会显示需重新输入，公开配置不丢失。
- 模型配置失败不妨碍查看历史岗位和已投状态。
- TXT/MD/DOCX/PDF 与粘贴文本都有提取诊断；异常文本不会直接冒充高质量解析，粘贴兜底可完成全流程。
- 画像包含可用于匹配的教育、经历、项目和技能，且“参与”不会升级为“主导”。
- AI/Python/解决方案多份简历可独立保存、打开、更新并被岗位匹配推荐到具体版本。
- 未保存筛选条件无法启动旧 Search Plan；无明确地点的通用用户必须主动选城市。

## 阶段 D：LLM 岗位事实、匹配与最终分池

问题范围：P-002、P-003、P-007、P-032、P-035、P-036、P-037、P-038、P-039、P-040、P-041。

实施内容：

1. 删除岗位分析阶段的空简历 `analyzeResume`；只读取已保存 CandidateProfile。
2. JobUnderstanding 仅输入平台事实和 JD，按来源内容哈希缓存，不依赖候选人。
3. MatchDecision 只输入候选人事实、真实简历版本、JobUnderstanding 和必要 JD 证据，不输入本地总分或关键词结论。
4. 缓存键加入各调用的 pipelineVersion；analysis 记录 profile/searchPlan/pipeline/source 版本并支持 stale。
5. 结果契约要求可核对证据；不完整允许一次修复，仍失败进入待分析。
6. 模型失败不能进入主投；保留岗位并提供重试。
7. 通用规则只负责链接、城市、确认过期和用户明确硬边界；技术主线、经验可冲和整体匹配交给模型。
8. 最终顺序固定为“硬边界 -> 语义分池 -> 双休/薪资/外包/风险/新鲜度同池排序”。

验收标准：

- 同一 JD 对不同候选人的 JobUnderstanding 完全复用且内容一致，MatchDecision 可以不同。
- 修改提示词/pipelineVersion 后旧缓存不会命中；只修改排序偏好不重跑 JobUnderstanding。
- 保存 Search Plan 后旧 analysis 明确 stale，不能展示成当前结论。
- 模型超时、非法 JSON 和空证据结果不会进入主投。
- 建立至少 30 条人工标注集，覆盖 AI 应用、Python 后端、Java/Go/C++ 主栈、算法训练、实习、产品/销售和信息不足；硬排除误放为 0，前排岗位必须有 JD 与简历双证据。
- 模型 apply/caution/review/skip 与最终主投/先聊/备选/不建议映射一致，页面显示主要决策来源。

## 阶段 E：按需沟通、结果跟进与诊断反馈

问题范围：P-045、P-046、P-047、P-050。

实施内容：

1. 从扫描和批量分析链路删除 `draftCommunication`，普通岗位使用 BOSS 通用招呼语。
2. 只有有证据的强推荐岗位提供按需定制招呼语按钮，不自动发送。
3. HR 回复入口接收 HR 原话；事实足够时返回 1-2 个版本，缺关键事实时只询问当前所需的一项并保存为 `user_provided`。
4. 增加无回复待跟进池；用户点击时按需生成一次跟进，不做无限自动提醒。
5. 推荐异常反馈成为独立事件，可与已投、保留、跳过并存；只进入诊断报告，不改变权重。

验收标准：

- 扫描普通岗位时 communication 模型调用数为 0。
- 强推荐点击后生成的文案必须包含该 JD 具体职责和候选人具体项目证据，且可复制、不可自动发送。
- GAP、离职、到岗等事实缺失时不会猜测；只询问一项，用户补充后相同问题可复用。
- 标记无回复后岗位进入待跟进池，不从系统中消失。
- 用户已投但报告“核心技术栈推荐错误”时，投递状态和诊断反馈都能保存。

## 阶段 F：调用指标、测试工程与最终验收

问题范围：P-048，以及前述所有阶段的回归与迁移。

实施内容：

1. 模型调用记录 kind、provider、model、cacheHit、latency、attempts、HTTP/错误码、usage 和 JSON mode 回退，不记录 Key、完整简历或完整 JD。
2. 修复 `flow_smoke` 子进程收尾，建立统一测试入口并保证失败码可靠。
3. 为数据库 schema 变化增加幂等迁移和现有数据回填；迁移前再次备份主库。
4. 完成全量自动化、主库只读一致性检查、真实 BOSS 小批扫描和页面人工验收。
5. 更新 README、操作说明和故障诊断，记录已知平台限制。

验收标准：

- 所有测试脚本可由一个命令运行并正常退出，重复运行两次结果一致。
- 日志能区分浏览器耗时、模型耗时和缓存命中，且敏感信息扫描为 0 命中。
- 主库迁移后 `quick_check=ok`，259 个既有唯一岗位和 33 个既有用户决策不少于基线，事件历史不丢。
- 真实小批扫描完成后，页面链接可打开、字段抽样一致、队列可分页、状态重启后保留。
- 无需 Codex 才能启动本地应用；项目自带启动脚本、Edge 连接方案和环境诊断能够独立工作。

## 完成规则

- 每阶段先补会失败的自动化检查，再修改实现，再跑该阶段与全量回归。
- 真实 BOSS 验收只在自动化通过后进行，控制频率并保存扫描日志。
- 某项因平台限制无法完成时，必须给出复现证据、降级行为和用户可见提示，不能静默跳过。
- 只有阶段 A-F 全部通过，才将目标标记为完成。
