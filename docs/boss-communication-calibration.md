# BOSS 沟通页面校准规程

状态：`pending`；`executionEnabled: false`。

本文是恢复真实页面访问后的人工校准操作规程，不是实时页面报告。当前仓库尚未验证真实 BOSS 页面、真实 DOM 选择器、活动聊天身份或点击后结果；在完成本文全部门槛前，不得把任何推断写成页面事实，也不得开启生产沟通执行。

## 1. 前置条件

1. 由操作者确认 BOSS 账号处于正常、未受限状态：没有登录失效、验证码/验证页、访问频控、异常重定向、账号冻结或明确的恢复时间。把检查时间、账号标识的脱敏值、页面 URL 和结果写入校准记录；不要保存 Cookie、Token、简历或未脱敏岗位信息。
2. 只使用同一个已登录 Edge profile，并准备两个固定标签页，标签名必须保持不变：
   - `BOSS-SEARCH`：搜索、列表和只读详情定位；不执行沟通。
   - `BOSS-COMMUNICATION`：未来沟通页面的只读检查和单岗位动作；不为岗位新建标签页。
3. 两个标签页不并行工作。一个岗位只能按“搜索定位 -> 沟通页检查 -> 结果记录 -> 回到搜索”顺序处理；任一步未完成、页面身份变化或出现风险信号，都先停止并记录。
4. 校准前只能运行离线测试、fake adapter 和本地脱敏 fixture。真实页面探针必须由用户主动恢复访问后单独安排；本规程没有证明当前页面已经可访问。

## 2. 必须收集的真实页面证据

每类至少保存一张脱敏截图和一份同一时刻的脱敏 DOM 快照。截图需能看出页面类型、岗位标题/公司脱敏值、关键控件及状态；DOM 快照需保留元素标签、可见文本、`id`/`name`/稳定属性、可见性和祖先关系。记录 URL、页面标题、岗位 ID、采集时间、证据文件哈希和探针序号。禁止保存个人头像、聊天内容、联系方式、Cookie、Token 或完整岗位隐私数据。

证据清单：

- `ready`：岗位身份与批次快照一致，岗位可用，页面明确出现可沟通动作及其精确可见标签；没有“继续沟通”等已完成状态。
- `already_communicated`：同一岗位身份下，页面明确显示已经沟通/继续沟通或等价的已沟通状态；只读确认，不点击。
- `unavailable`：同一岗位身份下，岗位关闭、下架、详情不可用或页面明确表示不能操作；不得把按钮缺失单独推断成不可用，须保留页面证据。
- `post-click detail state`：批准的单次校准点击之后，保存详情页变化、URL/岗位 ID、动作组件状态和时间顺序；“调用返回成功”本身不是证据。
- `active chat state`：进入聊天状态后，保存脱敏的聊天对象身份、岗位身份、页面/对话 URL 和可见状态；不得只凭聊天框存在就认定身份匹配。

每份证据必须能回答“这是哪个岗位、哪个账号、哪个页面状态、何时观察到”。缺任一项就标记为未校准，不能进入开启评审。

## 3. 只读探针顺序与节奏

严格按以下顺序执行，每一步完成记录后才能进入下一步：

1. 在 `BOSS-SEARCH` 检查当前 host、登录状态、风险提示和搜索结果身份；不要点击沟通动作。
2. 读取一个候选的稳定岗位 URL、岗位 ID、标题和公司快照，和本地批次快照交叉核对；不一致立即标记 `target_mismatch` 并停止该岗位。
3. 切换到 `BOSS-COMMUNICATION`，在同一 profile 中打开该岗位 URL；只读取页面类型、岗位身份、动作文本和现有沟通状态。
4. 先采集 `ready`、`already_communicated`、`unavailable` 三类证据；只有三类都能区分时，才安排后续的单次校准点击。
5. 每次只探针一个岗位。出现登录失效、验证页、频控、异常重定向、结构丢失、浏览器断连或结果不明确时，立即停止，不刷新重试、不换标签绕过、不切换账号。

使用现有策略的随机节奏：目录/列表导航等待 1.5-2.5 秒 / 2.5-4 秒，详情打开等待 3-5 秒，滚动等待 2.5-4 秒；连续 18-26 次页面动作后冷却 4-7 秒。详情探针每 6-8 次增加 15-25 秒微冷却，每 16-20 次增加 90-150 秒冷却。沟通动作即使未来开启，也遵守岗位间 15-20 秒随机间隔和 10 分钟 30、30 分钟 60、24 小时 150 的访问上限；失败不消耗重试预算来“撞”页面。所有操作仍以站点风险信号为最高优先级。

## 4. 选择器验收规则

选择器只有同时满足以下条件才可以进入候选清单；满足候选清单不等于允许执行：

- 页面必须是可信 BOSS host。当前代码层的岗位 URL 约束是 `https://www.zhipin.com/job_detail/<job-id>.html`；实际页面 host、协议和路径仍必须用真实页面证据重新记录，不能用 fixture 推断。
- DOM 查询结果必须唯一，且元素本身可见、在 viewport 内、未被其他元素覆盖；点击点用 `elementFromPoint` 复核，命中元素必须仍是目标元素或其受信任子节点。
- 可见标签必须与脱敏截图和 DOM 快照中的精确文本一致；模糊首匹配、相似文案、仅靠 class 顺序、仅靠坐标或仅靠 aria 猜测均不合格。
- 目标岗位 ID、标题、公司必须与批次不可变快照一致；页面导航、刷新、弹窗关闭或 DOM 变更后重新做全部身份检查。
- 点击前立即重新检查 URL、host、岗位 ID、唯一性、可见性、viewport、覆盖关系和精确标签；任一检查失败就不点击并记录 `action_unavailable` 或 `target_mismatch`。
- 选择器变更必须带新旧证据哈希、离线 fixture 回归结果和人工复核记录；不得通过改 selector 直接把 `executionEnabled` 改为 `true`。

## 5. 点击后的判定

点击最多派发一次，必须先写入审计事件并把该岗位置为 `click_dispatched`。随后只依据真实页面的可重复证据判定：

- `succeeded`：详情页/聊天页、岗位身份和沟通状态均与证据清单一致，且能在页面上确认动作已经完成。
- `already_communicated`：点击前或恢复后页面明确显示已沟通；不再派发第二次点击。
- `ambiguous`：点击已派发但页面、URL、对话身份或结果无法唯一确认，或浏览器在结果确认前断开。禁止自动补点、自动重试或把工具返回值当成功。

岗位身份不一致、岗位不可用、动作缺失或风险页出现时分别记录 `target_mismatch`、`job_unavailable`、`action_unavailable` 或风险停止原因。`ambiguous` 必须保留待人工解决，只有人工证据确认成功或明确停止后才允许终结。

## 6. 单次显式批准检查点

在真正校准点击前暂停，向用户展示：脱敏岗位标题/公司、岗位 ID、当前可信 URL、动作精确标签、前置/后置证据哈希、预计只点击一次。用户必须明确批准“本岗位本次校准点击”；批次确认、勾选列表、默认按钮、键盘回车或历史批准都不算批准。

批准只对当前岗位、当前页面状态和当前一次点击有效。发生导航、刷新、DOM 变化、超过 60 秒或切换标签后批准立即失效，必须重新做身份检查并重新请求批准。没有批准时，系统和操作者都不得调用真实点击 API。

## 7. BOSS 变化时的回滚与禁用

发现 DOM 改版、文案变化、host/URL 变化、身份无法确认、点击结果变化、频控或任何异常时：

1. 立即停止当前批次和所有 BOSS 页面动作，不刷新、不重试、不换账号、不绕过验证。
2. 保留已记录的 `click_dispatched`、`ambiguous` 和未完成项；未完成项标记为 `stopped`，不要删除审计记录或自动补点。
3. 将中央策略恢复为 `calibration.status: "pending"`、`executionEnabled: false`，撤销生产 `start` / `resume` 暴露；确认执行断言仍返回 `BOSS_COMMUNICATION_CALIBRATION_REQUIRED` / HTTP 409。
4. 隔离失效 selector、页面快照和截图，注明触发信号与最后一个可信证据哈希。先更新证据和离线 fake fixture，再重新走本文全部门槛。

## 8. 开启门槛与验收测试

只有以下条件全部满足，才可以在评审批准后把 `executionEnabled` 从 `false` 改为 `true`：

- 五类真实页面证据齐全且脱敏：`ready`、`already_communicated`、`unavailable`、点击后详情状态、活动聊天状态；每类均有截图、DOM、URL/岗位身份、时间戳和哈希。
- 账号未受限检查通过，两个固定标签使用同一 profile，串行锁、访问预算、随机节奏、断点和风险立即停止均有记录。
- 选择器验收通过：唯一、可见、在 viewport、未覆盖、精确标签、可信 host；岗位身份重检在点击前立即执行，并有 DOM 变化和不匹配的负例。
- 离线测试通过：状态机、幂等/单次点击、暂停/停止/恢复、歧义人工解决、批次选择校验、额度和 dashboard fail-closed 检查；生产 start/resume 在门槛关闭时必须在创建浏览器前拒绝。
- 经用户再次显式批准，完成一条真实校准点击并核对点击后详情和活动聊天证据；随后经用户批准完成三条以内的小批次试运行。任何歧义都阻止扩大批量。
- 评审记录明确批准人、证据哈希、代码/策略版本、回滚负责人和复验日期。没有完整记录时继续保持 `pending/false`。

本仓库当前只完成离线边界和规程定义，未完成上述真实页面证据、单次真实点击或小批次试运行。
