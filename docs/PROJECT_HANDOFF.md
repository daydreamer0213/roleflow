## 已确认的产品决策

### 搜索与匹配

- 生成模式的通用筛选只优先支持区域和公司规模；行业默认不限制。
- 继承模式保留用户在平台搜索页手工设置的原生条件，仅轮换关键词。
- 不建立庞大的职业模板库。模型根据确认的简历事实提出建议，用户可修改并确认。
- 搜索词分为目标岗位、相近岗位、业务场景、延伸尝试；A/B/C 只表示扫描频率。
- 技术栈是岗位匹配证据，不能默认作为无语义的单独搜索词。
- 岗位匹配必须区分人岗匹配与岗位质量；JD 的不合理愿望清单不能自动变成候选人的硬性不匹配。
- 多平台继承模式按平台分别保存：用户先在 BOSS、智联等各自手工设好搜索页，RoleFlow 只替换关键词；不跨平台翻译或放宽原生筛选。

### 沟通与进展

- 已发起沟通不等于已投递简历；后续实现必须把两者分开。
- 自动回复的第一阶段是“发现/粘贴消息、可靠关联岗位、生成草稿、用户确认”，不是自动发送。
- 缺少用户事实、岗位关联不确定、涉及敏感个人解释或面试邀约时，系统必须停在人工确认；面试邀约默认只提醒与记录，不生成建议回复或自动安排时间。
- 默认不保存聊天正文；只保存必要的结构化事件和用户确认的短摘要。

## 推荐实施顺序

不要把所有设计合成一个大改动。每次只选择一份实施计划，完成测试、审查、提交后再进入下一项：

1. 通用证据岗位匹配与匹配偏好卡；
2. 搜索意图组合；
3. 跨平台通用筛选；
4. 求职进展与人工确认工作流；
5. 多平台只读搜索与继承模板基础；
6. 在上述基础稳定且获得独立只读证据后，再接入首个真实平台；
7. 最后才讨论只读消息发现。

排序不是授权自动实施；每一项开始前仍由当前用户确认范围和优先级。

## 交接与验收协议

其他代理完成一个小任务后，交接信息必须包含：

1. 分支名与提交哈希；
2. 修改了哪些文件、为什么；
3. 实际运行的测试命令和完整结果；
4. 未覆盖的风险、未验证的真实页面前提或后续建议；
5. 是否读取/写入了真实平台或主项目数据（正常答案应为“否”）。

RoleFlow 的验收者应先读取该提交的 diff、相关测试和计划任务，再决定是否合并。不要因“模型或编辑器说完成”而跳过验证。
