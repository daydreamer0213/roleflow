# 通用证据岗位匹配设计

## 状态与范围

本设计已完成讨论，待用户审阅本文后再进入实现计划。实现仅在隔离 worktree 与 `codex/generic-evidence-matching-design` 分支中进行；不访问真实 BOSS、不启动 8787、不读取或写入主项目 `D:\Guo\ZhiPing\data\jobs.sqlite`。

本次解决的是“岗位匹配”模块，而不是搜索页筛选或 BOSS 沟通：让技术、运营、产品、设计等不同职业方向都能依据 JD 和简历中的实际证据得到可靠判断。

首轮验收以电商运营、内容/新媒体运营、用户运营为样本，同时保留技术岗的回归保护。

## 问题与目标

当前流程已有“简历结构化 -> JD 理解 -> 模型匹配 -> 本地规则保护”的骨架，但 JD 分类、关键词优先级、风险提示和部分本地评分明显以 AI 应用、Python/RAG、后端和算法候选人为默认对象。运营类岗位仅被粗略归入一个类别，不能把店铺运营、投放、GMV、内容增长、留存、CRM 等事实作为与技术栈同等的匹配证据。

目标不是预制大量职业模板，也不是让模型自由改写程序规则。目标是建立一个固定的、可审计的通用证据匹配内核：

1. 程序保留跨职业的底线和风险保护；
2. 模型在固定字段中理解每位用户的已证实经历与每份 JD 的实际要求；
3. 每个结论都必须给出 JD 与简历证据；
4. 模型不能以岗位不是技术岗为由扣分，也不能把 JD 的愿望清单误作全部硬门槛；
5. 用户在简历变化后确认新的个性化匹配设置，旧确认版本不会被悄悄替换。

## 不在本次范围内

- 不增加“电商运营模板”“内容运营模板”等生产规则分支；三个运营方向仅是验收样本。
- 不修改搜索筛选的城市、公司规模或行业方案。
- 不改变真实 BOSS 的读取、点击、沟通或投递行为。
- 不根据外部市场数据断言某个薪资是否客观偏低；只比较用户明确设置的薪资底线，以及 JD 内可观察到的职责膨胀或矛盾。
- 不采集、保存或生成 BOSS 聊天消息正文。

## 用户可见对象：匹配偏好卡

在现有 `CandidateProfile` 的事实基础上，新增一张用户可见、可编辑、须确认的“匹配偏好卡”（下称“卡”）。它不是另一份简历，也不是模型自行生效的策略。

卡包含：

| 字段 | 含义 | 约束 |
|---|---|---|
| 目标方向 | 用户希望重点匹配的工作方向 | 模型建议，用户可编辑确认 |
| 强证据 | 简历中已明确证明的职责、能力、成果或领域经历 | 每项必须能回溯到简历事实 |
| 可迁移能力 | 可用于相邻岗位的经验，例如淘宝运营到抖音电商运营 | 标为“可迁移”，不得伪装成已有平台经验 |
| 需谨慎转向 | 与用户已有经历相关但存在关键缺口的方向 | 仅作谨慎提示，不是自动跳过 |
| 用户补充偏好 | 用户主动添加或修改的目标、排除项和说明 | 优先于模型建议，保留修订记录 |

示例：一位有淘宝店铺、ROI 优化、活动复盘经历的用户，卡可以显示“电商运营/平台运营”“店铺活动、投放复盘、GMV 与转化数据”“数据分析、内容策划、跨部门协作”，并把“直播操盘”列为需谨慎转向，而非声称已有直播经验。

### 生命周期

1. 首次有简历时，模型生成草稿卡，用户查看、编辑并确认后才可用于扫描。
2. 之后若上传的简历内容哈希不变，复用已确认卡，不重复要求确认。
3. 内容哈希变化时，生成新的草稿卡；它不会自动替换当前已确认卡。
4. 在用户确认新卡前，界面明确显示当前扫描仍使用哪一版已确认简历和卡。首次没有已确认卡时不得扫描。
5. 用户确认新卡后，新扫描和新分析使用新版本；历史岗位保留原分析所用的卡版本与判断，不重写历史。
6. 用户手工编辑卡后保存为新的已确认修订版，后续扫描使用该修订版；模型不得在同一简历未变化时覆盖用户编辑。

## 通用岗位判断流程

每个岗位按以下顺序处理：

```text
用户确认的简历与匹配偏好卡
              +
             JD
              |
              v
    1. 通用基本条件与安全检查
              |
              v
    2. 模型拆解 JD 的真实工作与要求
              |
              v
    3. 模型逐项对照简历证据
              |
              v
    4. 独立判断岗位质量/风险
              |
              v
    5. 程序复核证据，输出四档建议
```

### 1. 通用基本条件与安全检查

此层不依赖职业类别。它处理用户明确的城市、薪资底线、全职/实习偏好、明确排除方向，以及已有的登录、风险、虚假或违规保护。

“硬性不符合”只能来自可核对的事实：例如用户要求上海全职而岗位为广州实习，薪资明确低于用户自己的最低接受值，或 JD 明确要求某项资格且候选人确定不具备。另一种例外是 JD 反复明确某项能力为该岗位唯一、不可替代的核心，且已确认简历和卡都没有任何相应证据；它可构成核心能力不符。岗位罗列许多平台、工具或“优先”事项时，不能把其中任一项默认升级为此例外。不得把“不是 Python/RAG”或“没有某个运营平台”当成默认硬性不符合。

### 2. JD 通用结构化

模型对 JD 只输出固定的通用字段：

| 字段 | 内容 |
|---|---|
| 核心工作 | 该岗位真正持续要做的职责，通常 1–5 项 |
| 核心要求 | 完成核心工作不可缺少的能力、方法、领域或工具 |
| 加分项 | “优先”“了解”“加分”等非必要条件 |
| 成果期望 | 目标、指标、交付物或责任结果 |
| 明确资格 | 明确届别、在校、学历、证书等资格条件 |
| JD 质量关注点 | 职责明显堆叠、前后矛盾、关键信息缺失等 |
| 风险信号 | 收费、培训贷、虚假或其他已有规则定义的风险 |
| JD 证据 | 对应字段的 JD 原文片段或可定位摘要 |

Python、GMV、作品集、Java、CRM、Figma 等均只是“核心要求”或“加分项”的候选内容；其重要性由 JD 的实际表述决定，而不是由内置职业词表决定。

### 3. 按证据匹配

模型将每个核心工作或核心要求与已确认简历/卡逐项对照，并只允许以下状态：

- `matched`：简历有直接证据；
- `transferable`：工作原理相近，但平台、行业或场景不同；
- `missing`：JD 明确为核心，简历明确没有相应证据；
- `unknown`：JD 或简历无法判断；
- `not_applicable`：加分项或与岗位主线无关的要求。

每项都必须同时保存 JD 证据和简历证据（或明确说明证据缺失）。模型不得把参与过项目写成主导，也不得把相邻经验写成完全具备。

`missing` 本身不自动等于 `skip`。只有它满足上一节的“唯一、不可替代的核心能力”标准时，程序才可将其作为硬性不符合；其余缺失应视为可迁移缺口或信息不足，得到 `caution` 或 `review`。

### 4. 岗位质量与人岗匹配分开

系统分别保留两个结论：

1. **人岗核心匹配**：用户是否有能力完成核心工作；
2. **岗位质量/风险**：JD 是否职责膨胀、信息矛盾、关键信息不足，或存在已有安全风险。

“要求十项全能但工资不高”不能自动证明候选人不匹配。若薪资仍在用户接受范围内，应标注为“职责可能膨胀”或“要求与待遇需确认”，作为岗位风险，而不是扣减候选人的核心匹配。只有薪资低于用户主动设置的底线，才是硬性跳过条件。

### 5. 四档建议的固定映射

| 人岗核心匹配 | 岗位质量/风险 | 证据状态 | 建议 |
|---|---|---|---|
| 核心职责明确匹配 | 正常 | JD 与简历证据充分 | 推荐投递 (`apply`) |
| 明确匹配 | 职责膨胀、JD 夸张或需注意的非安全风险 | 证据充分 | 谨慎投递 (`caution`) |
| 存在合理可迁移缺口 | 正常或一般 | 证据充分 | 谨慎投递 (`caution`) |
| 无法判断或证据不足 | 任意非明确风险状态 | JD/简历至少一侧不足 | 待确认 (`review`) |
| 明确硬性不符合 | 任意 | 有可核对证据 | 跳过 (`skip`) |
| 唯一、不可替代的核心能力明确缺失 | 任意 | 有 JD 与简历证据 | 跳过 (`skip`) |
| 诈骗、收费、违规或现有风险保护命中 | 任意 | 有风险证据 | 跳过 (`skip`) |

模型提出 `apply` 或 `caution` 时，程序必须检查其含有至少一条 JD 证据和至少一条简历证据。缺少任一侧证据时，降为 `review`。`skip` 必须提供可核对的硬性不符合或安全风险，不能仅以“经验较少”“缺少加分项”或“职位类别不同”跳过。

## 模型与规则的边界

模型负责理解自然语言、归纳证据、识别可迁移经验和 JD 质量关注点；程序负责字段契约、版本管理、硬条件、证据完整性、结果降级和安全保护。

模型不得：

- 生成或修改本地评分代码、阈值、职业词表或提示词；
- 自行放宽用户的城市、薪资、工作性质等明确偏好；
- 在缺少简历事实时创造技能、成果或项目职责；
- 将 JD 的加分项、愿望清单或不明确描述升级为硬性阻断。

这保留模型的个性化理解能力，同时让同一简历和 JD 在同一版本下可复核、可测试。

## 兼容与迁移

- 现有 `CandidateProfile` 和已上传简历继续保留。首次升级到本功能时，为当前简历生成一次草稿卡并要求确认；确认后不重复打扰用户。
- 历史岗位分析保持不变，并带有其当时的分析修订和卡版本。不得因新简历或新规则静默改写历史结论。
- 新实现以新的分析/契约版本参与缓存键和分析修订计算，保证旧模型缓存不会被误当成新通用判断结果。
- 现有技术岗的明确核心栈不被放松：当 JD 明确把 C++、Java/Spring 等作为单一核心能力且简历无证据时，仍可作为可核对的核心缺口；只是这些词不再是全局默认中心。

## 首轮验证与验收

测试只使用测试数据库、JD/简历 fixture、注入的假模型和页面 fixture。不得访问真实 BOSS 或主项目数据库。

### 运营验收 fixture

| 样本 | 预期 |
|---|---|
| 淘宝店铺、ROI 优化、活动复盘经历，对应电商运营核心职责 | 核心匹配；若 JD 正常且证据充分，可 `apply` |
| 同一候选人面对含天猫/抖音/直播/拍摄/剪辑等大而全要求的 JD | 识别核心与愿望清单；标职责膨胀，通常 `caution`，不得因缺少全部附加项 `skip` |
| 内容/新媒体经历面对用户运营岗位 | 有合理可迁移点时 `caution`，明确显示缺口 |
| 用户运营经历面对工作主线为纯销售的岗位 | 主线明确不符时 `skip`，且给出 JD 与简历证据 |
| JD 或简历不足以判断 | `review`，不得伪造匹配理由 |

### 必须保留的回归保护

- Python/AI 应用的简历与对应核心岗位仍能在证据充分时获得正确推荐。
- JD 明确将 C++、Java/Spring 等作为单一核心能力而简历无证据时，不能被模型随意放行。
- 城市、薪资底线、全职/实习、明确资格、虚假/收费/违规等原有保护继续生效。
- 有 `apply`/`caution` 但缺少任一侧证据的模型输出必须降为 `review`。
- 简历内容不变不产生新确认；内容变化产生草稿卡；草稿未确认不替换现用卡；历史分析不被重写。
