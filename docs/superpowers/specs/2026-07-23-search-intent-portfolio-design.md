# 搜索意图组合设计

## 状态

已确认设计，尚未实现。实现仅能在隔离 worktree 中使用测试数据库、模型假件和 DOM fixture 验证；不得访问真实 BOSS、主项目数据库或启动 8787。

## 问题与目标

当前 `SearchPlan` 使用一组平铺的 `keywords[{word, priority, reason}]`。`A/B/C` 同时承担了“这是什么词”和“多久扫描一次”的含义，导致用户看不出系统覆盖哪些岗位名称、哪些只是业务场景或可选延伸。遗留 mock 和关键词逻辑还对 Python、RAG、Agent 等技术词有历史偏向。

本设计把关键词改为平台无关的**搜索意图组合**（query portfolio）。它不建立职业模板库：模型只根据已确认的用户目标和简历证据提出候选词，用户可修改并确认。它应让技术、运营、产品、设计等方向都能用同一套结构。

## 范围

本期包含：

- 将搜索词标为目标岗位、相近岗位、业务场景、延伸尝试四类；
- 保留 A/B/C，但仅表示扫描优先级；
- 模型输出每个建议词的理由和简历/确认偏好依据；
- 在计划页显示、编辑、启停和解释各类词；
- 将已启用的词稳定映射到现有扫描、分析、统计和快照；
- 让扫描反馈能按四类词说明岗位供给。

本期不包含：

- 为任意职业编写固定岗位词典或行业模板；
- 根据外部招聘市场自动新增职业方向；
- 改变 BOSS 的真实访问、沟通、投递或频率控制；
- 实现第二个平台的关键词语法；平台适配器只在未来消费这里保存的通用语义。

## 搜索意图组合

每一项采用以下固定结构：

```json
{
  "id": "related-role-llm-application-engineer",
  "phrase": "大模型应用工程师",
  "category": "related_role",
  "priority": "B",
  "enabled": true,
  "source": "model",
  "reason": "职位名称不同，但工作主线通常与 AI 应用开发相近。",
  "evidence": ["已确认目标：AI 应用开发", "简历项目：RAG 知识库问答"]
}
```

字段语义如下：

| `category` | 用户界面名称 | 含义 |
|---|---|---|
| `target_role` | 目标岗位 | 用户明确希望重点投递的职位名 |
| `related_role` | 相近岗位 | 名称不同但核心工作通常相近的职位名 |
| `scenario` | 业务场景 | 以业务场景或工作内容命名、可补召回的搜索词 |
| `exploratory` | 延伸尝试 | 有可迁移证据但不能视为同等匹配的方向 |

`source` 只能是 `model`、`user` 或 `migrated`。`evidence` 是短证据标签，不是原始简历全文。`id` 由类别和标准化短语确定，同类别内不允许重复。全组合最多 18 项，避免一项计划无限扩大浏览与详情预算。

## AI 应用开发示例

候选人的真实证据为 Python、FastAPI、RAG、Agent、企业知识库落地；没有模型训练、微调或推理框架优化经历。

| 类别 | 默认搜索词 | 默认优先级 | 处理方式 |
|---|---|---|---|
| 目标岗位 | AI 应用工程师、LLM 应用工程师 | A | 每日优先覆盖 |
| 相近岗位 | 大模型应用工程师、智能体工程师、AI 产品工程师 | B | 每日补充覆盖 |
| 业务场景 | RAG 工程师、知识库问答、AI 工作流 | C | 广泛扫描；用户可提升为 B |
| 延伸尝试 | 算法工程师、NLP 工程师 | C，默认关闭或广泛扫描 | 结果默认不能因词命中而获得强推荐 |

Python、FastAPI、Redis、向量数据库等技术栈主要是**匹配证据**。它们只有在用户明确打开，或与场景组成可解释短语时，才成为搜索词；不能自动单独搜 `Redis`、`Python` 后把所有命中岗位视为目标岗位。

## 扫描语义

1. `target_role` 默认为 A，`related_role` 默认为 B；二者构成每日扫描的基础覆盖。
2. `scenario` 与 `exploratory` 默认为 C，进入广泛扫描。用户可以明确提升优先级，但界面保留类别说明。
3. `enabled=false` 的项不进入任何扫描、分析上下文或统计。
4. 现有 A/B/C 预算、薪资 lane、城市拆分、去重和低频串行规则保持不变；本设计不能以“场景词太多”为理由悄悄缩小已确认的覆盖目标。
5. 每轮快照保存实际使用的短语、类别、优先级和计划版本。结果统计按类别聚合，帮助解释“主目标岗位少”究竟来自城市、筛选条件还是词覆盖。

## 用户界面与确认

计划页用四个清晰区块替代单一逗号分隔关键词框。每一项显示短语、扫描优先级、理由、证据和开关。用户可以：

- 新增、删除或修改短语；
- 调整类别和 A/B/C；
- 关闭不想搜索的词；
- 把延伸尝试提升为主目标，或反向降级；
- 查看每日扫描与广泛扫描分别会使用哪些词。

模型建议只是一份草稿。首次生成、简历实质变化或用户主动“重新推荐”时显示变化；相同的已确认简历与计划不重复要求确认。若通用证据匹配卡已存在，模型可引用其已确认目标与证据，但不能借此新增未确认职业方向。

## 模型与程序边界

模型负责提出词、类别、理由和已有证据；程序负责：

- 契约校验、上限、去重和类别合法性；
- 禁止没有目标岗位或相近岗位的可执行计划；
- 保留用户手工词和开关，不被同一份简历的再次模型调用覆盖；
- 将所有下游调用统一转换为启用的短语列表；
- 维持“词命中不等于岗位强匹配”的边界。

模型不得凭“热门方向”猜测用户愿意转行，不得把简历中不存在的经历写成证据，不得把技术工具、加分项或 JD 愿望清单自动提升为用户的主目标。

## 兼容与迁移

既有 `keywords` 在读取时转换为 `migrated` 项：保留原词、A/B/C、理由和启用状态，类别默认为 `target_role` 或 `related_role`，由原 `directions` 与关键词是否相同决定。保存新计划后，旧 `keywords` 继续作为兼容投影，直到所有扫描、CLI 和分析入口都改用统一的 `planQueryItems()`。

历史扫描、岗位分析和报告不重写。新扫描只使用当时保存的意图组合快照。

## 验收

- AI 应用开发示例能把职位名、场景和算法/NLP 延伸尝试正确分层；
- 非技术 fixture 也能输出相同四类结构，且不因不含 Python/RAG 被降级；
- 每日扫描只使用启用的 A/B 项，广泛扫描包含启用的 C 项；
- 用户关闭或删除的项不进入 URL、分析上下文或统计；
- 旧计划无 `queryPortfolio` 时可无损读取并保持既有扫描行为；
- 所有验证只使用测试数据库、假模型和 fixture。
