# 求职进展与人工确认工作流设计

## 状态

已确认设计，尚未实现。此设计是未来“自动回复”功能的前置基础，不授权自动发送、真实消息轮询、真实 BOSS 点击或日历写入。

## 问题与目标

当前项目能从扫描结果生成主投、先聊确认和备选清单，能在用户确认后串行发起 BOSS 沟通，并在异常时停下。它也能生成首次招呼、粘贴 HR 原话后的回复和一次跟进文案。

但“沟通成功”目前会写成 `applicationStatus=applied`，这混合了“已发起聊天”和“已投递简历”两种不同事实；HR 的后续消息依赖用户手动粘贴，岗位、招聘方、聊天线程和约面进展没有统一载体。

本设计新增一张**求职进展卡**，让每个岗位机会都有可审计的当前阶段、事件时间线和下一步待办。它先服务于人工处理和安全文案辅助，再为未来的只读消息发现、多平台关联和面试安排提供稳定接口。

## 范围

本期包含：

- 将“已发起沟通”与“已投递简历”分开表达；
- 为一个候选人和一个岗位建立一张进展卡；
- 记录不含消息正文的结构化事件与人工待办；
- 将现有“粘贴 HR 原话 → 生成草稿”接入该卡；
- 提供人工标记回复已发送、邀约中、已约面、拒绝、关闭的操作；
- 继续使用用户确认的事实库，缺事实时只向用户提问。

本期不包含：

- 自动读取任意平台收件箱、自动识别聊天 DOM 或后台轮询；
- 自动点击发送、自动接受面试、自动创建日历事件；
- 保存 HR 原话、用户回复正文或整段聊天记录；
- 依据招聘方名称猜测岗位关联。

## 进展卡模型

一张卡绑定 `profileId + jobId`，并保存平台、岗位快照、可选招聘方身份、可选平台线程标识、当前阶段、下一步和最后事件时间。

```text
岗位：AI 应用工程师 / 某公司
平台：BOSS
招聘方：张女士（若平台可靠提供）
线程：平台线程 ID（若可靠提供）
阶段：等待回复
最近事件：已发起沟通
下一步：等待招聘方消息
```

阶段采用受控状态，而不是任意备注：

| 阶段 | 含义 |
|---|---|
| `contact_started` | 已可靠发起沟通，但尚未声明已投递简历 |
| `waiting_reply` | 已发起沟通或已回复，等待对方下一步 |
| `needs_user_action` | 收到问题、关联不明或缺少用户事实，必须人工处理 |
| `reply_ready` | 已生成草稿，等待用户复制、确认和手动发送 |
| `interview_invited` | 对方提出面试或时间确认，等待用户决定 |
| `interview_scheduled` | 用户确认了面试时间、形式或备注 |
| `resume_submitted` | 用户明确标记为已投递简历 |
| `rejected` | 用户或平台明确拒绝 |
| `closed` | 用户主动关闭机会 |

不是每个岗位都必须创建卡；扫描阶段和纯推荐阶段仍只保存岗位分析。首次人工动作、已验证发起沟通或用户记录进展时才创建卡。

## 事件、隐私与事实

进展卡的时间线只记录事件类型、发生时间、操作者、短摘要和必要元数据，例如 `contact_started`、`incoming_message_classified`、`reply_draft_created`、`reply_confirmed_sent`、`interview_invited`。

默认不保存以下内容：

- HR 原话、候选人回复正文、聊天截图和附件；
- 浏览器 cookie、账号、完整聊天线程 URL 中可能携带的敏感参数；
- 未经用户确认的个人解释。

HR 原话仅在用户粘贴的当前请求中交给模型分类和起草；日志、事件和数据库只保留类型、缺少事实的键和用户确认的短摘要。到岗时间、GAP、期望薪资等可复用信息仍由用户主动保存到既有事实库。

## 现有 BOSS 沟通的映射

| 现有验证结果 | 新进展事件 | 新阶段 | 对旧 `applicationStatus` 的处理 |
|---|---|---|---|
| `succeeded` | `contact_started` | `waiting_reply` | 不再把新事件写成“已投递” |
| `already_communicated` | `contact_already_exists` | `waiting_reply` | 不再把新事件写成“已投递” |
| `job_unavailable` | `job_unavailable` | 不建卡或关闭卡 | 继续标为岗位无效 |
| `target_mismatch` / `ambiguous` | `verification_needed` | `needs_user_action` | 保持人工复核，不自动推进 |
| 用户明确点击“已投递简历” | `resume_submitted` | `resume_submitted` | 继续使用 `applicationStatus=applied` |

迁移时，历史上带有 `communication_succeeded` 或 `already_communicated` 原因码的“已投”记录补建为 `waiting_reply` 卡，但不改写历史岗位状态。界面优先展示进展卡，所以用户看到的是“已发起沟通”，而不是误导性的“已投递”。

## 人工回复流程

以 AI 应用工程师岗位为例：

```text
用户粘贴：RAG 知识库项目实际已经上线使用了吗？
        ↓
系统确认当前岗位卡；没有可靠关联则停在“需人工关联”
        ↓
模型分类为“项目事实确认”，仅读取该岗位 JD、确认简历和用户事实
        ↓
有事实：生成最多两条可复制草稿，卡进入“草稿待确认”
缺事实：不生成回答，只询问“该项目是否实际投入使用？”
        ↓
用户复制并在平台手动发送后，点击“已手动发送”
        ↓
卡进入“等待回复”
```

对于“明天下午三点可以视频沟通吗？”这类内容，分类为 `interview_invitation`，进入 `interview_invited`。本期仅让用户确认或记录时间与形式；不访问外部日历，也不自行接受邀请。

## 用户界面

岗位详情中显示简洁进展区：当前阶段、最近安全摘要、下一步、事件时间线和人工操作。队列增加“等待回复”“需要处理”“邀约中/已约面”视图。已有的“已投”按钮改为“已投递简历”，避免与“已发起沟通”混淆。

草稿页明确提示“只生成到本页，不会自动发送”。关联不确定、事实不足、敏感问题和任何面试邀约都不能被自动发送或自动确认。

## 与未来能力的边界

未来可按以下顺序扩展，但不属于本期实施：

1. 只读、低频地发现平台的新消息元数据；
2. 用稳定线程 ID 和岗位 ID 自动关联，无法关联时停在人工处理；
3. 在充足审计样本和单独授权后，再评估低风险操作的半自动化；
4. 由用户明确授权后再接入日历，先做候选时间建议和冲突提示。

不论未来是否采用 OpenClaw 或其他通用自动化框架，平台适配、身份关联、事实约束和用户确认都保留在 RoleFlow 内。

## 验收

- BOSS 沟通成功显示“已发起沟通/等待回复”，不再等同于“已投递简历”；
- 每个岗位最多有一张活动进展卡，重复事件幂等；
- 缺少岗位关联或真实事实时不生成可发送回复；
- HR 原话与回复正文不落库、不写入审计日志；
- 面试邀约只能进入人工确认阶段；
- 既有沟通批次的暂停、异常中断、校准门和真实页面防护不被放宽；
- 测试只使用临时数据库、假模型和假浏览器。
