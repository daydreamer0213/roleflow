# 实施安全、升级兼容与验收门禁设计

## 状态

已确认设计，尚未实现。本文件约束已确认的通用证据匹配、搜索意图组合和求职进展工作流计划；它不改动现有主项目的真实求职流程。

## 目标

在开始这些计划前消除三类可预见问题：开发分支与正在使用的主项目分叉、旧用户升级后无法继续扫描、以及模型契约扩大后没有真实模型质量门禁。同步明确三轮工作流中的 C 类搜索词语义和进展卡的人工纠错方式。

## 共同前置条件

1. 主项目中正在验证的 BOSS 沟通修复必须先由主项目操作者在其工作树完成、运行相关测试并提交为明确 Git 哈希。优化工作不能读取、复制或合并主项目的未提交内容。
2. 新优化任务从该已提交基线创建隔离 worktree；开始前记录基线哈希，并检查 `git status --short`、相关文件差异和迁移版本。若基线中已包含 `communication_executor.js` 的修改，进展卡 Task 2 在该基线上实现，不得覆盖失败路径的证据记录。
3. 任何修改 `src/core/storage.js` 的任务不得并行实施。迁移版本使用目标基线的下一个连续编号（`vNext`），迁移测试以 `SCHEMA_VERSION` 和迁移名称校验；现有计划中的 `v5` 只描述当前 v4 基线上的预期，不是可跨分支硬编码的常量。
4. 不得访问真实 BOSS、其他真实招聘平台、`D:\Guo\ZhiPing\data\jobs.sqlite`，也不得启动或操作主会话的 8787。实现和测试只能使用临时数据库、假模型、DOM fixture 与假浏览器。

## 旧用户升级到匹配偏好卡

### 必须保持的体验

首次升级的旧用户即使没有更换简历，也必须能看到一次可编辑、可确认的匹配偏好卡；确认后，相同简历不再重复要求确认。用户不能靠修改文件内容来解锁扫描。

### 迁移规则

迁移不得调用模型或网络。它从每个已有候选人最新的 `profile_versions`、关联简历文档和保存的结构化画像生成一张 `source="migration"`、`status="draft"` 的匹配卡；仅当该候选人不存在任何 draft 或 confirmed 卡时创建。草稿必须保存原简历内容哈希和画像版本，并使用安全的确定性映射，不得捏造经历。

扫描仍要求 confirmed 卡。没有 confirmed 卡时，所有扫描入口必须返回 `MATCHING_CARD_CONFIRMATION_REQUIRED`，并提供对应 profile/card 的恢复入口；工作台直接打开草稿卡。CLI 不能重定向时，错误详情必须说明“在工作台确认现有匹配偏好卡”，而不是要求重新上传简历。

相同内容再次上传时先按 `profileId + resumeContentHash` 查询卡：

- 已确认卡：复用当前确认上下文并进入计划；
- 草稿卡：打开该草稿卡；
- 无卡：只为该已保存版本创建一次草稿，再打开它。

新内容才创建新的画像版本与草稿；旧 confirmed 卡继续用于扫描，直到用户确认新卡。

## 模型验收门禁

离线 smoke 和人工标注 fixture 是每次提交的必经检查。通用匹配合并前还必须有一次真实模型基准，但只能在用户对该次外部模型调用作出明确授权后运行。

真实模型基准使用脱敏的测试画像、人工标注 JD fixture 和临时数据库；不得默认读取真实简历、主项目数据库或 BOSS 页面。它在改造前和候选合并版本各留一份带提交哈希的结果，比较以下指标：

- `semanticStatus` 为 `failed`、`stale` 或 `pending` 的数量必须为 0；`partial` 只能进入人工复核，不得成为主投；
- 主投桶不得出现缺少 JD 与简历双方证据的岗位；
- 人工标注集的 recommendation、bucket、硬性误判数量不得差于改造前基线。

没有用户授权、没有有效模型配置或没有可比较基线时，代码可以完成离线验证和提交，但不得宣称已通过真实模型验收，也不得合并到正在使用的主项目。

## 搜索意图组合的 C 类词

`target_role` 与 `related_role` 的启用项分别默认 A、B；`scenario` 与 `exploratory` 默认 C。直接的每日扫描只使用 A/B，广泛扫描可使用 A/B/C。

三轮工作流采用更严格的供给扩张规则：第一、二轮只使用启用的 A/B；第三轮仅在原有“候选库存不足”条件满足时才加入启用的 C。用户把某项从 C 调为 B 后，该项才可进入前两轮。扫描快照必须记录该项为何在第三轮被选中。

## 进展卡纠错与重开

同一 `profile_id + job_id` 保持唯一进展卡，避免同一岗位分裂为多条互相矛盾的历史。不能通过删除或新建第二张卡纠错。

工作台提供人工“更正状态”操作：用户选择目标阶段并填写非空简短原因，系统保留原阶段和全部历史，新增 `manual_correction` 事件（包含 fromStage、toStage、用户摘要和时间），再转换当前卡。关闭的机会可由用户“重新打开”到 `needs_user_action`，同样记录事件；之后由用户选择正确阶段。任何纠错仍不读取或保存聊天正文，也不触发平台操作。

## 文档与计划一致性

每份计划列出的“修改”测试文件在开始实施时必须已存在；新建测试必须明确标为“Create”。计划引用不存在的测试入口时，应在交付前改为当前真实入口，或把它变成明确的新建文件。本轮已知的修正是：搜索意图计划使用 `semantic_pipeline_smoke.js` 和 `scan_snapshot_smoke.js`，进展卡计划使用 `semantic_pipeline_smoke.js`，而不是不存在的 `model_contract_smoke.js` 或 `job_analysis_cache_smoke.js`。

## 验收

- 主项目未提交改动不被优化分支读取或覆盖；任务报告中记录实际集成基线哈希。
- v4 旧数据库升级后，旧用户无需重新上传或修改简历即可打开一次草稿匹配卡并确认；相同哈希的后续上传不重复确认。
- C 类词不会进入前两轮；第三轮只有在候选不足时才使用它们。
- 手工错误标记为“已约面”后，可保留历史地更正或重新打开同一张卡。
- 真实模型门禁未获授权时不被运行；获授权后结果可比较、无敏感简历正文，且满足指定质量指标。
