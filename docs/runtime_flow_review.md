## R01 首次启动、模型配置与可用性验证

### 当前实际执行顺序

1. 用户打开项目首页 `/`。
2. 服务端读取 `.runtime/settings/model.json`，并检查 `.runtime/secrets/model-api-key.dpapi` 是否存在且文件大小大于 0。
3. 如果系统认为模型“未就绪”，首页跳转到 `/settings?required=1&next=/onboarding`。
4. 首次配置页面默认展示 DeepSeek，并预填基础地址、模型名和 30000 毫秒超时。
5. 用户点击“保存并进入简历解析”后，服务端先保存公开模型设置，再根据表单清除或保存 API Key。
6. API Key 使用 Windows DPAPI 按当前 Windows 用户加密，公开设置、SQLite 和日志不保存明文 Key。
7. 服务端再次判断模型是否“就绪”。对于真实模型，当前判断条件只有一个：加密 Key 文件存在且非空。
8. 判断通过后直接跳转到简历页面，不会请求模型厂商验证接口、Key、模型名或账户额度。
9. 用户上传简历并触发第一次真实模型调用时，才会发现鉴权失败、模型不存在、接口地址错误、余额不足或网络不可达。

### 已验证的正常部分

- DeepSeek、通义千问、OpenAI 和自定义兼容接口已有预设入口。
- API Key 使用 DPAPI 加密，现有测试确认 Key 明文不会出现在设置文件、加密文件和日志中。
- 模型设置和 Key 均保存在 `.runtime`，不会进入 Git 跟踪的公开配置。
- 同一厂商只修改模型名或超时时，API Key 留空可以保留原 Key。

### 本轮验证结果

- `tests/model_settings_smoke.js`：通过。
- `tests/model_settings_ui_smoke.js`：通过。
- UI 测试填写的是假 Key `ui-smoke-key-not-visible-after-save`，项目仍将其判定为“已配置”并跳转到简历页面。这证明当前测试只覆盖保存和保密，没有覆盖模型真实可用性。
- 当前本机状态：DeepSeek、`deepseek-v4-flash`、Key 文件存在；本轮没有发送真实模型请求，因此不能据此判断 Key、模型名或接口当前可用。

## R02-R07 简历、画像与搜索方案链路

### 当前实际执行顺序

1. 用户上传 TXT/MD/DOCX/PDF，或直接粘贴文本；项目先在本地提取文字，限制文件 5MB、文本 60000 字，少于 80 字判失败。
2. 提取成功后，把完整简历文本发送给所选模型，生成 `CandidateProfile`。
3. 再把 `CandidateProfile` 发送给模型，生成初始 `SearchPlan`。
4. 画像、提取文本、画像版本、一个初始简历版本和 Search Plan 在同一 SQLite 事务中保存。
5. 用户可分别编辑画像、简历版本元数据和筛选方案；扫描只读取数据库中已经保存的 Search Plan。
6. 首次真实 BOSS 扫描会预读站内筛选目录，再把薪资和经验映射为 URL 参数；已支持多个已知城市和多个薪资档位分 lane 扫描。

### 已验证的正常部分

- 上传文件与粘贴文本互斥，粘贴文本是稳定、低依赖的兜底；失败会记录提取方式、字数、预览、错误码和请求日志。
- `saveProfileAnalysis` 使用事务，第二次模型调用失败时不会留下半套画像与计划；这也确认了 P-005 的问题边界是“可恢复性差”，不是数据库半写。
- 手工编辑画像会新增 `profile_versions` 记录，不会直接覆盖而不留历史。
- 多选表单字段由 `parseBody` 保留为数组，城市、经验和薪资 lane 的多选提交本身没有丢值。
- BOSS 多城市扫描对内置城市码可逐城执行；3-5 年可冲条件确实会进入原生经验筛选，不是只在本地写了一个标签。

### 本轮确认的问题

- P-017：页面没有说明提取后的完整简历文本会发送给外部模型厂商。
- P-018：项目只检测 OCR 环境，没有任何 OCR 执行入口，却向用户提示可以使用本地 OCR。
- P-019：画像契约主动丢弃教育、工作/实习经历和个人优势等关键事实。
- P-020：所谓简历版本主要是元数据，新增文件不做语义解析，匹配时也不读取版本正文。
- P-021：筛选方案编辑和扫描是两个表单，未保存修改时会直接扫描旧方案。
- P-022：页面允许任意城市，但未知城市没有 BOSS 编码，直到启动扫描才报错。
- P-023：已预读的求职类型和学历筛选没有参与 URL 映射，实习岗位无法在源头排除。
- P-024：保存的薪资档位标签一旦与新目录不一致，会静默取消薪资原生筛选。
- P-025：本地提取只用字数判断成功，乱码、错序和 DOCX 页眉缺失仍会被当成可用简历。

## R08-R13 浏览器采集、去重、详情与规则门控

### 当前实际执行顺序

1. 页面启动后台 Node 子进程；子进程再连接 Edge Control 或项目专用 CDP，并自动挑选一个 BOSS 标签页。
2. 首次扫描预读 BOSS 原生筛选目录，按城市、关键词优先级和薪资 lane 串行打开搜索页。
3. 每个搜索页提取左侧卡片，尝试滚动追加；同一岗位在关键词和薪资 lane 内按 BOSS 岗位 ID 合并。
4. 当前实现只给配额内岗位导航独立详情页，随后却把所有岗位统一标成需要详情。
5. 本地规则先计算城市、活跃度、岗位类型、薪资、经验、工作制和技术栈标签；`blocked/refresh` 不进入模型，`ready` 才进入后续语义分析。
6. 模型分析完成后写入 `jobs` 当前快照和 `job_observations` 批次快照；同一岗位再次出现会更新当前快照并保留新 observation。

### 已验证的正常部分

- 2026-07-15 实测 Edge bridge、扩展和当前已登录 BOSS 搜索页均健康；浏览器采集是单标签串行，岗位抓完后本地模型分析才以默认并发 4 执行，没有多页面并发触发风控。
- BOSS 岗位链接会规范为稳定岗位 ID，跨关键词和跨批次的精确去重逻辑成立；重复出现不会覆盖“已投/跳过”等候选人状态。
- 未做任何状态标记的旧岗位原则上会继续留在 Plan 的待处理池，不会仅因“以前见过”就自动消失。
- 弱去重只把同公司、同标题、同地点标为 `possible_duplicate`，不会直接合并或删除，避免误伤不同招聘批次。
- 当前 `data/jobs.sqlite` 的 Plan #1 有 269 条 observation、259 个唯一岗位；薪资缺失 0、经验缺失 0、地点缺失 1。说明本批薪资/经验提取已基本正常，主要缺口不是这两个字段。
- 同一数据中活跃度缺失 227 条、详情未读 175 条；当前决策池 259 条中 201 条进入 refresh，和 P-004 的双栏/图标漏采结论一致。
- 对当前样例方案，明确异地、超过 3 日未活跃、实习、纯算法、明显非技术和无效链接作为硬边界是合理的；P-003 是否仍有大规模误杀，要在 P-004 修复后用同批岗位统计，不能用当前污染数据判断。

### 本轮确认的问题

- P-026：自动标签页选择不保证使用用户当前已登录页面，扫描前也没有连接与登录态预检。
- P-027：列表滚动选中了不可滚动的容器，导致很多关键词停在首批约 15 张卡片。
- P-028：长扫描缺少按搜索目标容错和中途保存，一个页面失败可能让整批已采集结果丢失。
- P-029：待处理池用 500 条 observation 做前置上限，重复记录可能把未处理岗位挤出页面。
- P-030：活跃度是历史快照但不会随时间老化，旧的“今日活跃”可能长期仍被当成 0 天。
- P-031：岗位详情变化哈希混入模型分析和分数，会把模型变化误报成 JD 变化。
- P-032：规则仍硬编码 AI/Python 偏好，不是根据通用用户画像和计划生成。
- P-033：经验判断会让 JD 正文中的任意高年限覆盖卡片的结构化经验字段。
- P-034：详情补读失败只写日志，不记录到岗位，用户无法知道失败原因和重试历史。

## R14 LLM 的 JD 理解和简历匹配

### 当前实际执行顺序

1. 只有本地门控结果为 `ready` 的岗位进入模型链路；`blocked/refresh` 直接生成规则结论。
2. 每个分析 runner 第一次处理岗位时，会把已保存画像作为 `profileHints`，额外调用一次 `analyzeResume({ resumeText: "" })`，得到本轮共用的候选人画像。
3. 每个岗位先调用 `understandJob`。实际输入不是纯 JD，而是完整岗位对象和候选人画像；完整岗位对象已经带有本地 `score/risks/qualityTags` 等派生判断。
4. 再调用 `matchJob`，输入候选人画像、简历版本元数据、完整岗位、JobUnderstanding 和本地 `ruleMatch`。
5. 当前还会继续调用 `draftCommunication`，然后把三次结果压成一个 `analysis`，最后再执行一次 `applyRuleGuard`。
6. 三类模型调用按“厂商 + 模型 + 调用类型 + 输入哈希”缓存；相同输入直接复用历史结果。

### 已验证的正常部分

- `understandJob` 提示词明确要求判断岗位真实主线、核心技术栈、资深工程要求、算法/训练、销售/运营和驻场等风险，不再只做关键词计数。
- `matchJob` 已明确要求结合完整 JD 和候选人证据，并专门限制 C++/Go、Java/Spring、重算法和资深高并发岗位不能因出现 RAG/Agent 就给高匹配。
- 模型输出经过统一契约规范化；推荐枚举、匹配等级、置信度和证据结构不会以任意形状直接进入页面。
- 浏览器采集结束后才在本地以默认并发 4 做模型分析，不会把模型并发变成 BOSS 多页面并发。
- 当前 Plan #1 的 259 个唯一岗位中，209 个停在活跃/详情待刷新、21 个规则跳过，只有 29 个最新快照保留真实模型结论。29 个中为 `caution` 21 个、`skip` 5 个、`review` 3 个，没有 `apply`。因此目前页面质量不能被理解为“模型已经完整分析了 259 个岗位”。

### 本轮确认的问题

- P-035：JobUnderstanding 口头上只理解 JD，实际却依赖候选人画像和本地分数，无法成为稳定、可复用的岗位事实层。
- P-036：模型缓存没有提示词、契约和分析管线版本，提示词修复后仍可能无限复用旧结果。
- P-037：语义匹配同时读取本地分数、风险和规则解释，容易被旧关键词判断锚定，不能独立纠正规则。
- P-038：真实模型失败时会按本地分数生成 `rule-fallback`，高分岗位仍可能看起来可投，页面没有把“模型分析失败”作为明显状态。
- P-039：保存 Search Plan 会重算本地分数，但不会重跑或作废旧语义结论，页面可能同时展示新分数和旧推荐。
- P-040：匹配契约允许理由、JD 证据和简历证据为空；当前 29 个真实模型最新结果中，2 个无匹配理由、6 个无 JD 证据、6 个无简历证据、10 个置信度为 0。
- P-048：模型链路只记录缓存命中和最终回退，缺少每类调用的耗时、重试次数、token 用量和厂商请求标识，难以判断成本与慢点。

## R15 推荐等级、风险排序、双休偏好与队列分池

### 当前实际执行顺序

1. `decisionState` 先按链接、城市、活跃度、详情完整性和规则硬边界，把岗位分为 `blocked/refresh/ready`。
2. `decisionBucket` 再把岗位放入 `primary/talk/backup/refresh/not_recommended`。
3. 真实模型的 `skip` 会进入不建议；除此之外，分池主要看本地 level、风险和质量标签，基本不使用模型的 `apply/caution/review`、匹配等级、证据或置信度。
4. 分池后依次按状态、分池、规则等级、硬风险、工作制、质量标签、风险数量、规则分数、活跃度和最近出现时间排序。

### 已验证的正常部分

- 双休是排序偏好而不是硬排除：双休优先，工作制未知仍保留，大小周/单休只降低次序。这与已经确认的投递策略一致。
- 2-3 年和低薪 3-5 年可冲岗位不会仅因年限被全局删除；明显异地、确认过期和明显非目标岗位仍可保留硬边界。
- 用户反馈目前只展示诊断提示，`penalty/bonus` 都是 0，不会自动学习并悄悄改变后续推荐，符合“先收集异常、不要把模型越改越糟”的决定。
- 当前 Plan #1 实际分池为：主投 0、先聊 3、备选 18、待刷新 201、不建议 37。这个结果主要受详情/活跃漏采和本地规则影响，不是纯语义模型排序结果。

### 本轮确认的问题

- P-041：最终分池只采纳真实模型的 `skip`，其余模型结论几乎不影响主投/先聊/备选，导致“语义模型负责匹配”在最终页面没有真正落地。
- P-039：本地重算和语义分析版本不一致时，排序读取新规则分，页面理由却可能还是旧模型结论。
- P-030：工作制排序可以保持不变，但活跃度必须先有时间有效期，否则旧的“今日活跃”会错误占据前排。

## R16 待投页面、岗位链接和用户状态标记

### 当前实际执行顺序

1. `/queue` 从当前 Plan 的全部历史 observation 中选每个岗位的最新记录，再叠加当前候选人的状态。
2. 只显示未处理、待确认和到期的保留岗位；已投、跳过、无回复、约面等状态从紧凑队列中移除。
3. 页面提供主投、先聊、备选、待读详情、活跃待核验、重复未投和不建议分组。
4. 每个分组过滤完成后直接 `.slice(0, 30)`；页面没有下一页或“加载更多”。
5. 卡片链接使用稳定 BOSS 岗位 URL 并在新标签页打开；状态操作写入候选人维度的当前状态，同时追加事件历史。

### 已验证的正常部分

- 岗位状态按“候选人 + 岗位”保存，同一岗位跨关键词、跨批次或跨 Search Plan 再次出现时，已投/跳过状态不会丢失。
- 当前数据库已有 27 个已投、6 个跳过，事件表也各自保留 27/6 条记录；不是只在页面内存中临时标记。
- 精确去重没有删除岗位，重复岗位仍可在“重复未投”查看；BOSS 岗位链接保存在岗位记录中。
- 主页面默认只展示主投和先聊，备选、刷新和不建议分开，方向上比把全部岗位压成一张高密度列表更合理。

### 本轮确认的问题

- P-042：每个紧凑分组最多展示 30 条且无分页。当前待读详情有 157 条、活跃待核验 41 条、不建议 37 条，超出的岗位从该页面无法继续浏览。
- P-043：“今日决策队列”读取的是整个 Plan 的历史待处理池，但卡片不展示首次/最近出现时间，也没有“本批新增 / 历史积压”主切换，用户无法直观看出今天抓到什么、昨天剩下什么。
- P-044：紧凑页面的“保留”按钮没有日期输入；保存空 `reviewAt` 后，队列逻辑会立即把该岗位再次视为到期，因此点击后不会真正离开当前队列。
- P-049：`renderQueuePage` 在 `return renderCompactQueuePage(...)` 后还保留一整段不可达的旧队列实现，容易让后续维护者误判真实页面逻辑。

## R17 强推荐定制沟通、无回复跟进与 HR 回答

### 当前实际执行顺序

1. 所有进入真实模型匹配的岗位都会尝试调用 `draftCommunication`，不区分强推荐、普通可沟通或备选。
2. `draftCommunication` 的系统提示只有“按 CommunicationDraft JSON 契约生成沟通草稿”，没有要求引用该岗位和候选人的具体证据，也没有约束 GAP、离职原因等信息来源。
3. 每张紧凑岗位卡都显示只读招呼语和复制按钮；数据库当前 259 个最新岗位快照全部有招呼语，存在 228 种文本。
4. 模型可能生成 `hrReplies`，但当前页面没有展示或调用入口。
5. 页面上的“HR 回复 / 跟进备注”只把用户输入保存成事件，不会调用模型生成回复；“无回复待跟进”也只改变状态，不生成跟进消息。

### 已验证的正常部分

- 项目不会自动点击 BOSS 沟通或自动投递，对外发送动作仍由用户确认。
- 模型沟通调用失败不会让整个岗位匹配失败，会保留已有招呼语继续展示。
- 用户输入的跟进备注会进入事件历史，后续查看时可以取最近一条。

### 本轮确认的问题

- P-045：扫描阶段为所有可分析岗位批量生成招呼语，和 D-001“普通岗位用 BOSS 通用招呼语、只有强推荐按需定制”完全相反，增加模型成本和页面噪声。
- P-046：系统没有真正的按需 HR 沟通入口。预生成的 `hrReplies` 页面不可见，跟进表单只记录文字；缺少事实时也没有按 D-005 只追问当前问题所需的一项信息。
- P-047：标记“无回复待跟进”后，岗位会退出紧凑队列，但紧凑页面没有无回复/待跟进分组，用户无法形成稳定的二次跟进清单。

## R18 投递结果、过期刷新和下一轮扫描复用

### 当前实际执行顺序

1. 用户可把岗位标成已投、跳过、待确认、保留、无回复、约面、拒绝、岗位无效或薪资不匹配。
2. 当前状态保存在 `candidate_job_states`，每次变化同时追加到 `candidate_job_events`，因此既有当前结果也有历史轨迹。
3. 下一轮扫描按稳定岗位 ID 合并岗位事实，再按候选人 ID 读取旧状态；重复扫描不会把已投岗位恢复成未处理。
4. 负面结果可附带原因码，聚合结果用于展示同公司/同关键词的诊断提示，但不自动修改规则权重。
5. 活跃度和详情刷新仍依赖历史 observation；超时、失败和旧快照问题沿用 R12/R13 的现有逻辑。

### 已验证的正常部分

- 状态隔离到候选人维度，同一份公共岗位数据可以被不同画像分别标记，不会互相污染。
- 已投、跳过、约面等是状态覆盖，事件历史仍保留，不会因后续改成拒绝就丢掉曾经已投的事实。
- 反馈没有自动改变推荐；现阶段保持为诊断台账是正确的，不需要先做不可控的在线学习系统。
- 未投岗位不会因为精确去重而被删除。真正可能让它暂时看不见的是 P-029 的查询上限、P-042 的页面截断或状态被误标，不是岗位 ID 去重本身。

### 本轮确认的问题

- P-030：活跃度快照不老化，旧岗位可能长期伪装成近期活跃。
- P-034：刷新失败没有岗位级错误和冷却，待刷新池可能反复处理同一失败岗位。
- P-043：下一轮扫描结果与历史积压没有清晰分层，用户会感觉每天都在重复同一批岗位。
- P-047：无回复状态没有专门待办池，结果闭环在“记录了状态”后中断。
- P-050：反馈原因和投递状态绑在同一个表单里，且只对跳过/无效/薪资不匹配保存；用户即使已投或保留但想报告“推荐异常”，原因也会被静默丢弃。

## 已确认的产品决策

### D-001 普通岗位不批量生成定制招呼语

- 普通“可沟通”岗位直接使用 BOSS 已配置的通用招呼语。
- 只有强烈推荐岗位才按 JD 和候选人证据生成真正有针对性的招呼语。
- 消息送达但未回复时，可以按需生成一次定制跟进。
- HR 已经提出具体问题时，再生成可复制回答。

### D-002 不增加重复确认负担

- 用户上传或粘贴的简历内容默认视为用户提供的事实。
- 不要求用户逐字段重复确认。
- 对模型提取错误提供轻量编辑入口；高风险内容在真正生成对外沟通时做边界约束。

### D-003 规则只负责边界，语义匹配交给模型

- 城市、无效链接、确认过期、明显非技术岗位等适合规则判断。
- 岗位实际方向、技术栈冲突、经验门槛是否可冲等不能只靠关键词命中决定。
- 规则门控的具体位置和宽严程度留到 R13 单独复盘。

### D-004 双栏卡片切换使用事件等待和短随机间隔

- 2026-07-15 实测使用的点击前后各 4 秒，仅用于人工确认页面行为，不作为正式扫描参数。
- 正式逻辑点击左卡后，应轮询右侧岗位 ID 或标题，确认右侧已经切换到目标岗位；切换成功后立即读取，不继续等待固定时长。
- 2026-07-15 最终节奏按动作分级：列表滚动 900-1600 毫秒、右栏切换 1000-2000 毫秒、列表/详情导航 1800-3200 毫秒、关键词切换 2500-4500 毫秒。
- 当前可见卡片由 DOM 一次性批量读取，等待发生在滚动、右栏切换和导航，不按每一张左卡机械停顿。
- 每随机 18-26 次外部页面动作增加一次 4-7 秒冷却；不再使用实际代码中曾存在的 15-40 秒长冷却。
- 右侧切换超时时只允许一次带退避的重试；再次失败就保留左卡数据、记录失败并继续，不反复点击同一岗位。
- 保持单标签页串行处理，不同时打开多个 BOSS 页面并行切换岗位。
- 右侧 `.job-detail-container` 是独立滚动区。先直接读取 DOM；只有完整 JD、招聘者活跃度或工作地址等必要字段缺失时，才滚动右栏触发懒加载并二次读取。
- 滚动只能作用于右侧详情容器，不能滚动页面主体或左侧列表；处理下一张卡前将右栏恢复到顶部，避免左右列表位置互相干扰。

### D-005 沟通口径不在简历解析阶段猜测

- 简历解析只生成简历可以直接支持的候选人事实、技能和项目贡献边界。
- GAP 原因、短期项目背景、离职原因、到岗时间等通常不在简历中，首次解析时不要求模型生成完整对外回答。
- 用户不需要在首次使用时逐项填写或确认这些问题；只有 HR 实际问到时，系统才检查是否已有用户提供的事实。
- 缺少必要事实时只询问当前问题所需的一项信息；用户回答后保存为 `user_provided`，后续相同问题直接复用，避免重复确认。
- 模型可以润色用户事实，但不能把模型推断保存为用户事实，也不能用通用模板补成个人经历。

### D-006 工作地点由用户决定，简历地点只用于预填

- 简历明确写了求职城市或目标工作地点时，模型可以提取并作为筛选页预选值。
- 简历只写了当前居住地、学校或经历所在地时，不能自动等同于求职城市。
- 简历没有明确求职地点时，Search Plan 保持地点未选择，由用户在开始扫描前选择；不能给所有用户默认广州。
- 当前用户的材料明确包含“广州优先”，因此可以预填广州，但这属于用户事实，不是系统全局默认值。

### D-007 不增加强制画像确认页，方案重生成必须由用户触发

- 简历解析完成后继续直接进入筛选方案页，顶部展示姓名、城市、目标方向、技能和项目摘要。
- 用户只有发现提取错误或希望调整方向时才进入画像编辑，不要求逐字段重复确认。
- 用户修改画像后，项目不能静默覆盖已经人工调整过的 Search Plan。
- 项目应明确提示当前方案是否仍基于旧画像，并提供“根据最新画像生成新建议”；只有用户主动点击后才创建或替换建议。

### D-008 岗位扫描只使用已保存画像，不重新解析空简历

- 用户上传新简历或主动点击“重新解析简历”时，才允许调用 `analyzeResume` 生成新画像。
- 正式扫描、详情补读和批次重新评估直接读取数据库中的 CandidateProfile。
- 用户手工修改后的技能、项目边界和求职方向必须原样参与岗位匹配，不能在扫描时被模型重新表述。
- 已保存画像不存在或无法读取时，应明确停止语义匹配并提示修复画像，不能用空简历临时生成替代画像。

### D-009 API Key 按模型厂商隔离保存

- DeepSeek、通义千问、OpenAI 和自定义接口分别使用独立的 DPAPI 密钥槽位。
- 同一厂商内部切换模型名时可以复用该厂商已验证的 Key。
- 跨厂商切换时只读取目标厂商的 Key；目标厂商尚未配置时明确要求填写，不能静默复用当前厂商密钥。
- 设置页分别显示当前厂商的“未配置 / 已保存未验证 / 已验证 / 验证失败”状态。
- 自定义接口的 Key 至少与其 Base URL 绑定；Base URL 改变后必须重新验证，不能默认旧 Key 继续适用。
- 所有 Key 继续使用 Windows DPAPI 加密，不写入公开配置、SQLite、日志或发布包。

## 当前问题台账

### P-001 模型配置被当成扫描的全局前置条件

- 优先级：高。
- 状态：已逐项说明并经用户确认。
- 当前逻辑：`POST /api/scan` 在连接 Edge 之前先检查 `modelReady`；Key 缺失、模型服务暂时不可用或配置状态异常时，整个扫描直接终止。
- 具体场景：已有画像和 Search Plan，DeepSeek 临时不可用；浏览器登录、BOSS 搜索、左卡采集和本地字段解析本来不依赖模型，但用户当天连原始岗位都无法保存。
- 修改方向：基础采集与模型分析解耦；模型不可用时继续采集和落库，岗位进入“待语义分析”，模型恢复后再补充 JD 理解和简历匹配。

### P-002 单个岗位的模型调用链偏长

- 优先级：高。
- 状态：已逐项说明并经用户确认；当前只确认删除批量沟通调用，JD 理解和匹配是否合并留到 R14。
- 当前逻辑：每个进入模型的缓存未命中岗位，依次调用 `understandJob → matchJob → draftCommunication`；`draftCommunication` 不检查最终推荐等级，谨慎或跳过岗位也会生成沟通文案。
- 规模示例：50 个全新岗位最多产生 1 次临时画像处理和 150 次岗位模型调用，其中 50 次沟通草稿大多不会使用。
- 修改方向：所有合格岗位保留 JD 理解和简历匹配；只有强推荐岗位在用户触发时生成定制招呼语，消息送达未回复时按需生成跟进，HR 提问时再生成具体回答。

### P-003 本地规则可能让大量岗位永远进不了语义匹配

- 状态：暂缓确认。2026-07-15 实测发现，上游双栏采集缺失会主动制造大量 `detail_unverified` 和 `activity_unverified`；必须先解决 P-004，再重新判断规则门控本身是否过严。
- 当前已知：只有规则标记为 `ready` 的岗位才进入 LLM；`blocked` 和 `refresh` 不进入模型。
- 不能直接下结论的原因：无效链接、明确异地等规则拦截合理；大量“待刷新”更可能是采集器没有读取右侧详情，而不是规则误杀。
- 后续：P-004 处理并用同一批岗位重跑后，再统计每种门控标签的数量和误判案例。

### P-004 BOSS 双栏详情没有被采集，绝大多数岗位被标记为详情缺失

- 优先级：高。
- 状态：2026-07-15 已通过已登录 Edge 实测确认。
- 页面真实结构：搜索页左侧是可滚动岗位卡片；点击左卡后，右侧 `.job-detail-container` 原地切换对应岗位详情，搜索页 URL 不离开 `/web/geek/jobs`。
- 当前代码：`collectCards()` 只读取左卡；收集结束后，`readDetail()` 将同一标签页导航到独立 `job_detail` URL。项目没有“点击左卡并读取右侧详情”的路径。
- 左卡可稳定取得：岗位 ID/链接、标题、薪资、经验、学历、公司、细分地点，以及招聘者在线图标。
- 右侧可稳定取得：完整 JD、职位技能标签、标题、薪资、城市、经验、学历、招聘者姓名、招聘者公司/角色、明确活跃文字和工作地址。
- 实测样例：从广东睿程智造科技切换到 CLPS 华钦软件后，左侧 active 岗位 ID 从 `671a6e91...` 变为 `b07695f3...`；页面仍停留在搜索页，右侧返回 `Java / AI / Python`、完整职位描述、`潘女士`、`刚刚活跃`、`CLPS华钦软件 · HR`。
- 右栏滚动实测：`.job-detail-container` 的可视高度为 546、内容高度为 1701，`overflow-y: auto`；当前样例的 635 字 JD 已完整存在于 DOM，说明“视觉上需要滚动”不等于“必须滚动后才能读取”。页面同时存在“查看更多信息”控件，后续需要防止把公司扩展信息误当成 JD 展开按钮。
- 活跃度漏读的具体原因：左卡在线状态主要表现为 `.boss-online-icon`，当前代码只在卡片文字中匹配“今日活跃/在线”等文本，因此会把有在线图标的岗位记为活跃未知；右侧则直接提供 `.boss-active-time` 文本。
- 扩大影响的具体逻辑：扫描结束时所有候选岗位都被统一设置 `detailRequired: true`，但只有详情配额选中的少量岗位会导航读取；其余岗位成为 `detailRead: false`，随后进入 `detail_unverified → refresh`。
- 修改方向：在搜索页内按岗位 ID 去重，顺序点击左侧新卡，等待右侧岗位 ID/标题切换后合并左右字段；不再把独立详情页导航作为主流程。
- 频率控制：按 D-004 的动作分级随机间隔和随机分段冷却执行，不沿用固定 4 秒等待，也不恢复 15-40 秒长冷却。
- 风控约束：单标签页串行操作，不并行点击；只点岗位卡片，不点“立即沟通”；切换失败时保留左卡数据并记录失败，不连续重试页面。

#### P-004 后续修改拆分

1. 左卡提取补全：岗位 ID、标题、薪资、经验、学历、公司、细分地点、技能文本和 `.boss-online-icon` 状态。
2. 双栏切换：点击左卡后校验 `.job-card-wrap.active` 和右侧岗位身份均对应目标岗位。
3. 右侧提取：先直接读取完整 JD、`.job-label-list`、招聘者姓名、`.boss-active-time`、公司/角色和工作地址；若必要字段缺失，再分段滚动 `.job-detail-container`、等待短随机间隔并二次提取，最后恢复 `scrollTop=0`。
4. 字段合并：左卡提供稳定元数据，右侧覆盖更完整字段；只有右侧读取成功才设置 `detailRead: true`。
5. 状态修正：删除“所有候选岗位统一 `detailRequired: true`”的处理，避免没有被选中导航的岗位自动进入待刷新。
6. 降级路径：右侧切换失败时保存左卡并记录明确失败原因；独立 `job_detail` 导航只作为少量人工补读或兼容兜底。
7. 节奏与日志：记录点击目标岗位 ID、右侧确认耗时、随机等待、重试次数和失败类型，但不记录 Cookie 或 JD 全文。
8. 小批量验收：先连续读取 10-20 张卡，核对字段完整率、错位率、平均耗时和页面异常，再决定正式批次参数。

### P-005 搜索计划失败会导致已生成画像一起丢失

- 优先级：中。
- 状态：已确认；保留两次任务，不要求强行合并。
- 当前逻辑：第一次用原始简历生成 CandidateProfile；第二次只读取 CandidateProfile 生成 Search Plan，不再读取原始简历；两步都成功后才统一保存。
- 具体场景：画像生成成功，但 Search Plan 请求超时或 JSON 校验失败；整个上传请求失败，已生成画像不落库，用户重试时需要再次上传并重复第一次模型调用。
- 附带风险：第一次如果漏掉地点、薪资或方向，第二次无法回看原始简历纠正，只会基于残缺画像或程序默认值继续推荐。
- 修改方向：第一次成功后先保存画像；第二次独立生成并保存 Search Plan。第二次失败时保留画像，允许只重试搜索建议，或先进入基础可编辑方案。

### P-006 首次链路可能跳过画像检查但又保存了多类对象

- 状态：已关闭，不作为缺陷修改。
- 复盘结果：简历上传后虽然直接进入筛选方案页，但页面已经展示姓名、城市、目标方向、技能和项目摘要，并提供“编辑画像”“重新解析简历”“管理简历版本”入口。

### P-007 已有画像在岗位分析阶段可能被再次模型化处理

- 优先级：高。
- 状态：已确认。
- 当前逻辑：每次创建岗位分析器后，第一次分析岗位时都会调用 `analyzeResume({ resumeText: "", profileHints: configs.candidateProfile })`；同一运行内通过 Promise 复用一次，但跨正式扫描、详情补读和批次重评不会复用。
- 具体问题：数据库画像已经完成结构校验，也可能经过用户人工修正；扫描时却用空简历让模型根据画像再生成一份临时画像，岗位匹配使用临时结果，而不是数据库原值。
- 用户风险：项目贡献边界、技能或方向可能在不同批次被重新措辞；页面画像不变，但实际匹配依据发生漂移，用户无法察觉。
- 运行风险：这次调用没有进入模型缓存；一旦失败，同一运行中的所有岗位都会从模型分析降级为规则结果。
- 修改方向：按 D-008 删除岗位分析阶段的 `analyzeResume` 调用，直接使用 `configs.candidateProfile`；同步补充测试，确保扫描、补读和重评都不会触发简历解析。

### P-008 “已配置”被误当成“模型可用”

- 优先级：高。
- 状态：已逐项说明并经用户确认。
- 具体逻辑：`isModelReady()` 对真实模型只检查加密 Key 文件是否存在，不验证文件能否解密，也不向模型接口发送测试请求。
- 具体场景：用户输错一个字符的 Key，点击保存后仍会进入简历页；直到上传简历、完成本地提取并等待模型响应时，才收到鉴权错误。
- 用户成本：一次配置错误会被推迟到更晚、更重的操作中才暴露，用户容易误以为是简历解析失败。
- 修改方向：保存后发送一次最小连接请求，并区分“已保存未验证 / 验证成功 / Key 无效 / 模型不存在 / 限流或余额不足 / 网络失败 / 密钥不可解密”；验证失败保留用户输入并停留在设置页，不再拖到简历解析阶段暴露。

### P-009 切换模型厂商时会静默复用上一个厂商的 Key

- 优先级：高。
- 状态：已确认。
- 具体逻辑：项目只有一个固定密钥槽位 `model-api-key`，不是每个厂商各自保存一个 Key。
- 具体场景：先配置 DeepSeek；之后切换成通义千问，API Key 输入框显示“已保存，留空保持不变”。用户留空保存后，项目会把原来的 DeepSeek Key 发给通义接口。
- 用户成本：页面仍显示“已加密保存”并允许继续，但第一次真实调用一定会鉴权失败，而且页面没有告诉用户当前保留的是哪家厂商的 Key。
- 修改方向：按 D-009 使用 `model-api-key-deepseek`、`model-api-key-qwen`、`model-api-key-openai` 和自定义接口密钥槽位；厂商内切模型复用，跨厂商只读取目标厂商密钥，并结合 P-008 分别记录连接验证状态。

### P-010 DPAPI 密钥文件不可解密时仍可能显示“已保存”

- 优先级：中。
- 状态：已确认；这是由 Windows DPAPI 的用户/机器绑定特性和当前“只看文件是否存在”的判断共同导致，不需要破坏现有密钥做迁移实测才能成立。
- 具体逻辑：页面只检查 `.dpapi` 文件存在且非空，不尝试解密。
- 具体场景：用户把包含 `.runtime` 的项目目录复制到另一台电脑或另一个 Windows 用户下；旧密钥文件仍存在，但 Windows DPAPI 无法用新用户解密。
- 用户成本：设置页显示 Key 已保存，真正提交简历时才出现“本机加密密钥库操作失败”。
- 修改方向：读取设置状态时验证密钥可解密，并区分“未填写 / 已保存且可用 / 文件存在但不可解密”；绿色发布包继续排除 `.runtime`，迁移后明确要求重新填写 Key。

### P-011 模型公开设置和 API Key 不是一次性保存

- 优先级：中。
- 状态：已确认；保存顺序和缺少回滚可由代码直接确定，故障注入只用于后续回归测试。
- 具体逻辑：服务端先写 `.runtime/settings/model.json`，再调用 DPAPI 保存 Key；两个动作之间没有回滚。
- 具体场景：用户切换厂商并输入新 Key，公开设置保存成功，但 PowerShell 或 DPAPI 保存失败。页面提示保存失败，实际厂商和模型已经改变，旧 Key 仍可能保留。
- 用户成本：用户以为本次修改完全失败，实际项目处于“新模型设置 + 旧密钥”的混合状态。
- 修改方向：结合 P-008、P-009，先把目标厂商 Key 保存到该厂商独立密钥槽并完成最小连接验证，最后才切换公开的活动厂商/模型设置；如果公开设置写入失败，旧活动配置仍有效，多出一个尚未启用的目标厂商 Key 不影响运行。补充 DPAPI 失败和设置写入失败的故障注入测试。

### P-012 预设模型名单过于固定

- 优先级：中。
- 状态：已对照 2026-07-15 厂商官方文档确认。
- 具体逻辑：DeepSeek、通义千问和 OpenAI 预设只能从代码内写死的模型名中选择；只有切换成“自定义兼容接口”才能手工填写模型名。
- 核对结果：DeepSeek 当前预设 `deepseek-v4-pro / deepseek-v4-flash` 与官方一致；通义官方当前推荐 `qwen3.7-max / qwen3.7-plus / qwen3.6-flash`，项目仍展示 `qwen-max / qwen-plus / qwen-flash` 泛化别名；OpenAI 项目预设仍以 GPT-4.1/4o mini 为主，无法反映账号当前实际可用模型。
- 具体场景：厂商新增、改名或下线模型后，用户不能在原厂商预设下直接填写新模型名，必须改用自定义接口并重新处理基础地址。即使旧别名仍兼容，用户也无法知道它实际指向哪个版本。
- 用户成本：预设本来为了减少配置工作，过期后反而会制造“地址正确但模型不存在”的错误。
- 修改方向：首次配置只保留少量经验证的推荐模型；Key 验证成功后优先调用厂商兼容的 `/models` 获取该账号可用列表，失败时回退内置预设；始终允许在当前厂商下展开“其他模型名”手工输入，不强迫用户切换到自定义接口。
- 官方依据：[DeepSeek Models API](https://api-docs.deepseek.com/api/list-models/)、[阿里云百炼推荐模型](https://help.aliyun.com/en/model-studio/models)、[OpenAI Models API](https://developers.openai.com/api/reference/resources/models)。

- 修改方向：至少对 `apply/caution` 要求一条 JD 证据、一条候选人证据和一条具体理由；`skip/review` 至少要求 JD 风险或信息不足说明。输出不完整时允许一次针对性修复请求，仍失败则进入“待分析”，不能用通用理由冒充完成。

### P-041 最终分池只采纳模型的 skip，忽略模型的其他语义结论

- 优先级：最高（直接决定岗位质量）。
- 状态：已确认。
- 当前逻辑：`decisionBucket` 只有在真实模型 recommendation 为 `skip` 时直接不建议；模型 `apply/caution/review`、fitLevel、confidence 和证据不会决定主投/先聊/备选，最终仍主要按本地 level 和标签分池。
- 具体场景：模型明确判定“信息不足，需要 review”仍可能因本地 level 进入备选；反过来，模型给出有证据的高匹配也可能因关键词风险标签留在备选。当前 29 个真实模型岗位没有一个进入 `apply`，页面也没有独立说明是模型结论还是规则分池。
- 用户影响：用户看到的是“模型理由 + 规则分池”的混合结果，推荐标签和理由可能不一致；升级模型也未必改善最重要的前排质量。
- 修改方向：固定三层融合顺序：先用通用硬边界拦截；再用有证据的 MatchDecision 决定主投/先聊/备选/不建议；最后用双休、外包、薪资弹性、风险数量和新鲜度只做同池排序。每张卡显示主要决策来源，避免隐藏分数汤。

### P-042 紧凑队列每个分组只显示前 30 条，没有分页

- 优先级：高。
- 状态：已通过代码和数据库实证确认。
- 当前逻辑：分组过滤后直接 `.slice(0, 30)`，没有 `page/cursor`、加载更多或跳转到同条件完整列表。
- 数据证据：当前待操作岗位中，待读详情 157 条、活跃待核验 41 条、不建议 37 条；对应页面分别只能看到前 30 条，剩余岗位没有继续浏览入口。
- 用户影响：岗位仍在数据库，却表现得像“筛选后丢了”；这会和 P-029 一起加重“有些没投的岗位是不是被去重掉了”的不信任。
- 修改方向：先修 P-029，让 SQL 对唯一岗位分页；紧凑队列增加最简单的上一页/下一页或加载更多，并显示“当前 1-30 / 共 N 条”。计数始终基于全量分组，不因当前页变化。

### P-043 “今日队列”混合当前批次与历史积压，卡片不显示时间来源

- 优先级：高。
- 状态：已确认。
- 当前逻辑：`/queue` 默认读取整个 Search Plan 的最新岗位快照，不限定当天或最新主扫描批次；卡片不显示 firstSeenAt、lastSeenAt 或最新所属批次。
- 数据证据：当前 259 个岗位的最新快照来自批次 #42（13 条）、#43（244 条）和详情刷新 #46（2 条），但页面统一标题为“今日决策队列”。
- 用户影响：用户无法区分今天新抓、今天重复出现、昨天未处理和仅做过详情刷新的岗位，容易误以为搜索链接没更新或每天只返回同一批。
- 修改方向：保留历史积压，不删除；主入口分成“本轮新增”“本轮重复”“历史未处理”三个可见范围，每张卡显示首次发现和最近出现。页面标题使用“当前待处理”或明确标注对应扫描批次，而不是无条件叫今日。

### P-044 紧凑页面的“保留”操作不会真正把岗位移出当前队列

- 优先级：中。
- 状态：已确认。
- 当前逻辑：紧凑卡片提交 `status=later`，但没有 `reviewAt` 输入；`compactAwaitingAction` 把 reviewAt 为空的 later 立即视为到期，所以页面刷新后岗位仍在。
- 具体场景：用户点“保留”希望稍后再看，状态文字变成保留，但岗位继续占据同一队列位置，操作看起来失效。
- 用户影响：增加重复处理和疑惑，也无法建立真正的备忘队列。
- 修改方向：紧凑按钮改成语义明确的“7 天后再看”，服务端默认写入 7 天后的日期；详细页面保留原生日期输入供调整。若需要永久收藏，应另设“收藏/备选”状态，不用空日期冒充稍后。

### P-045 扫描阶段批量生成招呼语，与已确认沟通策略相反

- 优先级：高。
- 状态：已确认。
- 当前逻辑：每个 `ready` 岗位都会调用 `draftCommunication`；系统提示只有一句契约说明。紧凑卡片无论主投、先聊还是备选都显示招呼语和复制按钮。
- 数据证据：当前 259 个最新岗位快照全部有非空招呼语，存在 228 种文本；模型缓存中也已有批量 communication 结果，但用户实际海投主要使用 BOSS 自带通用招呼语。
- 用户影响：每个岗位多一次模型调用，扫描更慢、成本更高；大量轻微改写的通用话术没有实际价值，反而增加阅读和选择负担。
- 修改方向：从扫描链路删除 `draftCommunication`。普通岗位不在页面生成话术；只有有证据的强推荐岗位提供“生成定制招呼语”按钮，用户点击后才调用模型，并要求引用该 JD 的具体职责和候选人的具体项目。始终只复制，不自动发送。

### P-046 当前没有真正可用的按需 HR 回复与跟进生成

- 优先级：高。
- 状态：已确认。
- 当前逻辑：模型预生成的 `hrReplies` 被存入 analysis，但页面从不显示；“HR 回复 / 跟进备注”只调用 `addFollowUpNote` 保存文字；“无回复待跟进”只改状态。没有接收 HR 原话并生成可复制回复的接口。
- 具体场景：HR 问“为什么 GAP 一年”，用户把问题贴进备注框后，系统只记录这句话，不会返回答案；若沿用预生成 `riskMessaging/hrReplies`，又可能使用简历中不存在的 GAP 或离职原因。
- 用户影响：项目在最能降低沟通消耗的环节没有完成闭环；已有模型调用既花了成本又不可用。
- 修改方向：新增按需沟通动作，输入当前 HR 原话，读取已确认简历事实和已有 `user_provided` 沟通事实。事实足够时生成 1-2 个可复制版本；缺少关键事实时只询问当前问题所需的一项信息，用户回答后保存并复用。不能在首次解析阶段预填 GAP、离职和短期项目原因。

### P-047 “无回复待跟进”状态没有对应的待办分组

- 优先级：中高。
- 状态：已确认。
- 当前逻辑：标记 `no_reply` 后，`compactAwaitingAction` 不再返回该岗位；紧凑页面没有 no_reply 标签页。完整 `/jobs` 虽能按状态过滤，但紧凑导航也没有明显入口。
- 具体场景：用户投递后数日标记无回复，希望稍后发一次针对性消息；岗位立即从日常队列消失，只能手工修改 URL 或进入旧完整页面寻找。
- 用户影响：结果被记录了，但没有下一步动作；大量无回复岗位无法形成低成本跟进列表。
- 修改方向：增加“待跟进”池，展示最后沟通时间、岗位链接和备注；用户主动点击时才生成一次定制跟进。只做提醒和文案，不自动发送，也不默认无限追踪。

### P-048 模型调用缺少可定位成本和慢点的运行指标

- 优先级：中。
- 状态：已确认。
- 当前逻辑：模型 adapter 解析正文后丢弃响应中的 usage 和请求标识；日志只有 cache hit/save、最终 fallback 和错误，没有单次调用耗时、实际重试次数、token 用量或 JSON mode 回退次数。
- 具体场景：一次扫描很慢时，只能看到最终完成或失败，无法判断是 JobUnderstanding 慢、MatchDecision 慢、批量招呼语浪费时间，还是厂商 429 重试。
- 用户影响：无法量化“为什么抓取后还要处理很久”，也无法比较 DeepSeek、通义或其他模型的真实成本和稳定性。
- 修改方向：记录不含敏感正文的调用指标：requestId、jobId、kind、provider、model、cacheHit、latencyMs、attempts、HTTP 状态/错误码、input/output token（厂商返回时）和 JSON mode 回退。API Key、完整简历、完整 JD 和模型原始输出不写普通日志。

### P-049 队列入口保留了 return 之后永远不会执行的旧实现

- 优先级：低（清理项）。
- 状态：已确认。
- 当前逻辑：`renderQueuePage` 返回 `renderCompactQueuePage` 后，仍保留旧的 `listDecisionQueue + renderDashboard` 代码块；该代码永远不可达。
- 具体场景：维护者看到旧代码里的“每次 15 条、最新批次”等逻辑，容易误以为当前页面仍按它执行，排查时得出错误结论。
- 用户影响：不直接影响运行，但持续增加“设计文档、代码表象和真实执行不一致”的维护成本。
- 修改方向：删除 return 后的不可达分支；`/queue` 只保留紧凑实现，`/jobs` 继续作为完整历史页。测试直接断言真实入口的分页和分池行为。

### P-050 推荐异常反馈与投递状态强绑定，部分反馈会被静默丢弃

- 优先级：中。
- 状态：已确认。
- 当前逻辑：同一个表单同时放“已投/待确认/保留/跳过”和不匹配原因；存储层只对 `skipped/invalid/salary_mismatch` 保存 reasonCode，其他状态即使提交了原因也会清空。
- 具体场景：用户认为岗位可以试投，但发现系统把“核心 Go + AI 工具调用”错误推荐为高匹配，于是选择“岗位方向不匹配”后点已投；投递状态保存了，诊断反馈却无提示地消失。
- 用户影响：开发阶段收集不到最有价值的“推荐错了但用户仍尝试”的样本；用户也不知道自己的反馈是否生效。
- 修改方向：把“报告推荐问题”做成独立轻量事件，可与已投、保留或跳过并存，包含原因码和可选一句备注；明确提示已记录。它只进入诊断报告和人工调优样本，不自动改变排序权重。

## 整改前全链路复盘结论

### 整改前产品判断

项目已经具备可继续打磨的核心骨架：简历入口、画像与 Search Plan、BOSS 原生筛选、Edge 只读采集、稳定岗位 ID、批次 observation、候选人状态、结果事件、工作制软排序和诊断日志都已存在。但当前还不能算稳定合格的通用岗位筛选产品，主要不是缺更多功能，而是四条关键链路尚未闭合：

1. 来源事实不完整：双栏、滚动、活跃图标和长任务容错让 201/259 个岗位停在 refresh。
2. 语义决策不一致：模型理解、规则分数、缓存和最终分池是多套版本，模型没有真正控制前排质量。
3. 队列可见性不足：查询前置截断、每池 30 条和历史/本轮混合，会让数据库中的岗位表现得像丢失或重复。
4. 沟通能力放错位置：扫描时批量生成无价值招呼语，真正需要的按需 HR 回复和无回复跟进反而没有入口。

### 集中修改顺序

1. **先修来源事实**：P-004、P-026、P-027、P-028、P-030、P-034。没有可靠卡片、右栏和活跃数据，后面的模型评估都没有验收意义。
2. **再修数据可见性**：P-029、P-031、P-033、P-042、P-043、P-044。保证岗位不被查询或页面截断，且新旧批次可解释。
3. **统一语义决策**：P-007、P-019、P-020、P-032、P-035 至 P-041。形成“岗位事实 -> 候选人匹配 -> 硬边界 -> 偏好排序”的单一链路。
4. **重做按需沟通**：P-045、P-046、P-047。删除扫描期批量文案，只在强推荐、无回复和 HR 已提问时调用模型。
5. **补齐通用化与工程边界**：P-008 至 P-013、P-017、P-018、P-021 至 P-025、P-048 至 P-050。最后处理设置、隐私说明、平台配置、观测和低风险清理。

### 明确保留的设计

- BOSS 浏览器操作保持单标签串行，使用事件等待、短随机扰动和分段冷却，不做多标签并发冒险。
- 精确岗位 ID 去重、历史 observation 和候选人维度状态继续保留；重复出现是信号，不是删除理由。
- 双休继续作为排序偏好，不作为一刀切硬条件；外包、2-3 年和低薪 3-5 年可冲也继续保留弹性。
- API Key 继续使用本机 DPAPI 加密并与公开设置分离，但要修复厂商隔离、可解密检查和保存原子性。
- 用户反馈继续只用于诊断和人工调优，不做自动在线学习。
- 用户上传或粘贴的简历默认视为已提供事实，不增加逐字段强制确认页。

### 集中修改后的实测门槛

1. 用真实模型完成首次配置、连通性检测、简历上传/粘贴、画像生成、方案编辑和保存重进；任何失败都能在页面看到具体阶段与 requestId。
2. 在已登录 Edge 上用 2-3 个关键词做小批真实扫描，验证列表实际滚动追加、左卡切换右栏、长 JD 滚动补读、活动图标识别和随机节奏；全过程不自动沟通。
3. 对同一批岗位保存抓取前后样本，人工核对标题、公司、地点、薪资、经验、活跃度、完整 JD 和稳定链接；只有页面本身未提供的字段才能记为未知。
4. 建立至少 30 条人工标注岗位集，覆盖 AI 应用、Python 后端、Java/Go/C++ 主栈、算法训练、实习、产品/销售和信息不足，分别验收 JobUnderstanding、MatchDecision、硬边界和最终分池。
5. 修改画像、Search Plan、提示词和模型后重复分析同一岗位，必须能明确看到缓存失效或 analysis stale，不允许继续展示版本不明的旧结论。
6. 构造超过 30 条和超过 500 条 observation 的数据，验证唯一岗位分页、各分组总数、本轮新增/重复/历史积压以及未投岗位不会消失。
7. 走完已投、保留到期、无回复、约面、拒绝、无效和薪资不匹配状态；跨关键词、跨批次和重启服务后状态与事件历史都保持一致。
8. 验证普通岗位不调用沟通模型；强推荐点击后才生成定制招呼语；无回复点击后才生成一次跟进；HR 问题缺事实时只询问当前所需的一项信息。
9. 注入模型超时、非法 JSON、429、Edge 未连接、未登录页、右栏切换超时和单关键词白屏，确认已采集岗位不丢、失败可重试、错误可定位且不会泄露 API Key 或完整简历正文。

## 复盘纪律

- 每次只处理一个编号环节，不跨环节直接设计整套新流程。
- 先记录真实代码和实测行为，再判断问题；用户观察与代码推断分开标记。
- “待验证”不能直接进入修改清单。
- 暂定方向不是最终方案，必须等相邻环节一起复盘后再定。
- 全链路复盘结束后，统一整理为：删除项、简化项、修复项、保留项和实测清单。
