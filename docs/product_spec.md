# 简历驱动投递筛选助手产品规格

## 定位

本项目不是岗位搜索器，也不是单纯的关键词过滤器。

目标是做一个简历驱动的投递决策助手：输入候选人简历、求职目标和招聘岗位，系统输出是否建议投递、为什么、用哪版简历、主推哪个项目、怎么和 HR 沟通，并把投递反馈沉淀下来。

## 当前用户

- 候选人：以当前上传并确认的简历画像为准。
- 地点：广州优先。
- 方向：AI 应用开发、Python 后端、RAG、Agent。
- 薪资：9-14K，可按岗位调整为 8-12K / 9-13K。
- 核心材料：德勤 AI 应用实习、DocMind、KnowledgeFlow、公开信息采集与智能日报生成工具。

## 核心原则

1. LLM 是核心分析层，不是文案装饰。
2. 规则层只做硬边界和安全兜底。
3. 所有对外口径必须稳健、可追问、不过度包装。
4. 投递动作必须人工确认，系统不自动沟通、不自动发送消息。
5. 分析结果必须可解释，最好能追溯到简历证据或 JD 片段。

## 分层架构

### 1. 采集层

职责：

- 从 BOSS 等招聘平台读取岗位列表和详情。
- 保留岗位链接、公司、岗位、薪资、地点、经验、活跃度、JD 原文。
- 标记岗位来源、关键词、抓取批次、重复出现情况。

不负责：

- 不判断深层语义匹配。
- 不自动点击立即沟通。
- 不自动发送消息。

### 2. 规则硬筛层

职责：

- 城市：广州优先，非目标城市降权。
- 薪资：过低、过高或明显不匹配时标记风险。
- 年限：0-3 年、1-3 年、经验不限优先；3-5 年结合薪资判断可冲。
- 活跃度：3 日内活跃优先，长期不活跃降权。
- 明显排除：销售、培训、讲师、课程顾问、纯前端、纯算法训练、纯运维、模糊外包。
- 状态过滤：已投、已跳过、重复出现、HR 状态。

边界：

- 规则层不负责语义判断。
- 正则和关键词只是 fallback，不应该伪装成智能匹配。

### 3. LLM 简历理解层

职责：

- 从简历原文拆解结构化画像。
- 提取技能、项目、经历、证据点、风险点、可投方向。
- 生成简历版本定位。
- 产出稳健对外口径。

典型输出：

- CandidateProfile
- ResumeVersion[]
- ProjectEvidence[]
- RiskMessaging
- SearchStrategySeed

### 4. LLM JD 理解层

职责：

- 判断岗位真实方向。
- 识别 JD 中的核心职责、技术栈、业务场景、隐性门槛。
- 识别伪 AI、培训/销售伪装、外包驻场、过期岗位、资深岗误投风险。
- 把 JD 拆成结构化 JobUnderstanding。

典型输出：

- realRoleType
- coreRequirements
- niceToHave
- hiddenRisks
- senioritySignal
- businessScenario
- evidenceSnippets

### 5. LLM JD-简历匹配层

职责：

- 结合 CandidateProfile 和 JobUnderstanding 判断匹配度。
- 推荐是否投递：强投 / 可投 / 谨慎 / 不投。
- 推荐简历版本。
- 推荐主推项目。
- 输出匹配理由、缺口、风险追问、面试前置说明。

典型输出：

- recommendation
- fitLevel
- fitReasons
- missingPoints
- riskQuestions
- recommendedResumeVersion
- primaryProjects
- greetingAngle
- confidence

### 6. 沟通助手层

职责：

- 根据岗位和匹配结果生成招呼语。
- 根据 HR 问题生成可复制回复。
- 处理 GAP、实习、项目来源、薪资、到岗、短期项目等高频问题。
- 保持口径稳健，不夸大项目职责。

### 7. 投递 CRM 与反馈层

职责：

- 记录已投、已跳过、已沟通、约面、拒绝、无回复。
- 记录岗位重复出现和历史判断。
- 统计关键词、岗位类型、简历版本的反馈表现。
- 反向优化搜索策略和排序。

## 决策流

```text
简历原文
  -> LLM 简历理解
  -> 候选人画像 / 简历版本 / 风险口径

岗位 JD
  -> 规则硬筛
  -> LLM JD 理解
  -> LLM JD-简历匹配
  -> 报告决策卡
  -> 人工确认投递
  -> CRM 反馈
```

## 报告目标

报告不只是岗位列表，而是投递决策台。

每个岗位至少展示：

- 推荐等级。
- 推荐动作：投 / 谨慎投 / 不投 / 先人工复核。
- 匹配理由。
- 风险点。
- 推荐简历版本。
- 主推项目。
- 招呼语。
- 岗位链接。
- 历史状态：首次出现 / 重复出现 / 已投 / 已跳过。

## 阶段计划

### 阶段 0：规格与契约

- 明确产品定位。
- 固化 LLM 分析层和规则层边界。
- 定义结构化数据契约。

### 阶段 1：画像与关键词规划骨架

- 候选人画像配置。
- 简历版本配置。
- 关键词规划。
- mock/rule 版匹配解释字段。

### 阶段 2：LLM Adapter 与结构化输出

- 增加 model config。
- 增加 OpenAI / DeepSeek / 通义 adapter 边界。
- 先支持 mock 和一种真实模型。
- 强制 JSON 输出和 schema 校验。

### 阶段 3：JD-简历匹配接入报告

- 每个岗位生成 MatchDecision。
- 报告展示推荐简历版本、主推项目、风险追问、招呼语角度。
- 缓存 LLM 分析结果，避免重复花费。

### 阶段 4：投递 CRM 闭环

- 已投、已跳过、已沟通、约面、拒绝。
- 今日新增 / 历史重复 / 已处理分区。
- 关键词和岗位类型反馈统计。

### 阶段 5：绿色交付

- installer 检测环境。
- portable Node。
- 专用 Edge profile。
- 打包 zip，排除登录态。

## 非目标

- 不做自动投递。
- 不做自动发送消息。
- 不做伪装浏览器行为绕风控。
- 不做大而全招聘平台系统。
- 不把模型建议当最终事实，用户保留最终判断权。
