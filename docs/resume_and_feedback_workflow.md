# 简历、质量与反馈工作流

## 简历入口

完成模型配置后，系统提供并列的三种入口：

- 上传 TXT、MD、DOCX 或带文本层的 PDF。
- 直接粘贴已有简历文本。
- 使用内置文本模板后按需填写。

系统先在本地提取文本，再把文本交给已配置模型做结构化解析。用户提交的文件文本、粘贴文本和模板文本都视为用户提供的事实材料；模型只能整理，不能编造公司、时间、职责、项目归属、学历或技能。

当 PDF 可提取文字不足时，系统会标记为疑似扫描件。OCR 只作为可选诊断信息：系统不会自动执行或下载 OCR 二进制工具。此时直接粘贴文本仍是稳定兜底。

## 模型设置

首次打开会先进入网页“配置模型”。选择 DeepSeek、通义千问或 OpenAI 预设后，接口基础地址和模型列表会立即带入，填写 API Key 并保存即可进入简历解析；未完成配置时不能进入正常投递流程。也可选择“自定义兼容接口”接入其他 OpenAI-compatible 服务。

API Key 不写入配置文件、SQLite、日志或发布包。项目使用 Windows DPAPI 按当前用户加密保存到 `.runtime/secrets`，扫描子进程会在本机内存中读取同一份设置。切换到另一台电脑时需要重新填写 Key。

## 模板内容

内置模板包含：

- 求职意向：目标岗位、城市和期望薪资。
- 教育经历：学校、专业、学历和时间。
- 工作、实习或协作经历：组织、岗位、时间、实际参与、结果和贡献边界。
- 项目经历：背景、本人工作、工具、结果证据和不应夸大的边界。
- 专业技能：按实际熟悉程度分组。
- 个人优势，以及可选的语言、证书和作品链接。

## 低打扰画像

解析后的画像和筛选方案默认可以直接使用。编辑完全按需；只有筛选所必需的信息缺失、文本提取失败，或用户主动想改方向、薪资、城市、技能和项目表述时，才需要介入。

系统会在内部保留输入来源元数据，服务于日志和模型护栏，但不会要求用户逐项确认。对外沟通只使用用户材料支持的内容；无法确定时使用稳健表达，而不是把推断写成事实。

## 简历版本

每个简历版本独立保存原始文件、解析诊断、适用方向、关键词、主推项目、使用说明和启用状态。只有启用版本会参与后续匹配和推荐。

## 岗位质量与反馈

岗位观察保留内容哈希和时间，可标记疑似同公司同岗位重复、详情变更和超过 14 天未再次出现的岗位。反馈按候选人画像隔离，支持已投、跳过、无回复、复核、稍后、约面、拒绝、无效和薪资不匹配。

约面会提升相关关键词和简历版本优先级；多次无效或薪资不匹配会降低优先级。所有数据只保存到本地 SQLite，系统不会自动投递或发送消息。

## 回归检查

常规检查、UI 流程检查和离线扫描检查应在发布前运行。UI 流程检查覆盖上传、粘贴文本、画像编辑、简历版本、筛选方案、离线扫描、约面标记和结果筛选。
