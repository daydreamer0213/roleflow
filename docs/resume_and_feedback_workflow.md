# 简历版本、岗位质量与反馈

## 简历版本

每次成功解析都会先保存：

- 原文件名、内容哈希、文本长度、提取方式和质量诊断。
- 该次独立的结构化画像分析。
- 适用方向、关键词、主推项目和启用状态。

后续岗位匹配推荐具体简历版本，不只引用当前画像快照。画像、简历版本、JD 内容、模型或分析契约变化时，旧匹配结果失效并重新分析。

## 岗位唯一性与快照

- BOSS job detail ID 是首选稳定来源 ID。
- 同一来源岗位只保留一条主记录，每次扫描写入独立观测。
- 薪资、年限、招聘方活跃度、正文和分析结果保留快照与内容哈希。
- 同公司同岗位弱重复只做提示和排序治理，不合并用户决策。
- 多轮未再次出现或活跃信息过期会进入待刷新，不直接删除。

历史未投岗位不会因跨关键词去重或新一轮扫描消失。已投、跳过、无回复、约面等状态属于候选人和方案，不随岗位详情刷新被覆盖。

## 投递状态

支持：

- 待确认、稍后处理、已投、跳过。
- 无回复、约面、拒绝、岗位无效、薪资不匹配。
- 跟进备注和复查日期。

`无回复` 有独立队列。只有用户点击时才生成一次跟进文案，不自动提醒或发送。

## 推荐反馈

推荐反馈与投递状态并存，原因包括岗位方向、核心技术栈、年限、薪资、地点、公司、活跃度、工作制和其他说明。

反馈的当前用途是诊断：

- 发现解析、详情读取、模型理解或排序中的异常。
- 为开发阶段复盘提供具体样本。
- 汇总常见误判，但不自动改写关键词、评分或公司权重。

不实现自动在线学习。用户反馈可能稀疏、带情绪或只适用于单个岗位，直接反哺排序容易越调越偏。只有经过人工复盘、标注集验证和回归测试后，才把稳定规律写入产品规则。

## 沟通事实

定制招呼语只对完整 JD、双证据齐全的主投岗位开放。HR 回复和跟进按需生成。

如果 HR 问 GAP、离职原因、到岗时间或短期项目，而本地没有用户提供事实，模型只能返回一个 `missingFact` 问题，不能同时生成可发送回复。用户回答后事实以 `user_provided` 来源保存，再生成稳健口径。

所有文案只供复制，不会自动发送。
