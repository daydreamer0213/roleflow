# 每日筛选与沟通工作流

本文描述 RoleFlow 当前的主使用路径。参数权威来源是 `src/core/product_policy.js`，状态规划逻辑位于 `src/core/workflow_run.js`。

## 一轮任务

```text
用户点击“执行一轮”
  -> 计算当天成功数、可用候选、候选缺口和剩余预算
  -> 库存不足时选择关键词并扫描；库存足够时跳过扫描
  -> 本地并发完成岗位理解与简历匹配
  -> 生成默认勾选的沟通清单
  -> 用户取消不想沟通的岗位并确认清单
  -> 用户点击“开始沟通”
  -> 串行打开岗位、核验身份、单次点击并核验结果
  -> 保存成功、已沟通、失效、歧义和未访问替补状态
```

RoleFlow 不在后台定时运行。每一轮都由用户主动启动，外部沟通还需要用户确认清单并再次点击开始。

## 每日目标与轮次

- 每日成功沟通目标：70 个。
- 每轮成功目标上限：40 个。
- 每天最多创建 3 轮；前 2 轮是主要工作轮次。
- 第一轮通常目标为 35；第二轮根据当天剩余目标动态计算。
- 第三轮只在可用候选少于 30，或本轮候选缺口至少为 8 时追加扫描；库存够用时不会为了凑数再扫。
- 两次扫描开始时间至少间隔 2 小时。无需扫描、直接使用库存的轮次不消耗网页访问预算。

轮次上限是防止同一天反复测试和重复抓取，不要求用户必须跑满 3 轮。

## 工作流预算

工作流每日共享预算：

| 项目 | 每轮上限 | 每日上限 |
| --- | ---: | ---: |
| 右栏详情读取 | 120 | 360 |
| 搜索页导航 | 20 | 60 |

已结束的扫描按实际访问量结算；正在创建、扫描或分析的轮次暂时预占完整计划预算。预算按北京时间自然日统计，跨日后重新计算。

高级入口中的独立“日常扫描”仍使用单次 `240` 个详情和 `40` 次页面预算，它是诊断与维护入口，不等同于上表的主工作流预算。

BOSS 访问控制还有独立硬上限和短窗口冷却。正常模式下右栏详情最多为 10 分钟 45 次、1 小时 240 次、24 小时 360 次；达到短窗口时等待，发现登录、验证或风控异常时立即停止。

## 搜索条件与关键词

启动扫描时，如果当前已登录 Edge 正停在有效的 BOSS 搜索结果页，RoleFlow 会继承该页面的城市、区域、薪资、经验、学历、行业等 URL 参数，只删除页码并逐次替换 `query` 关键词。这样用户可以先在 BOSS 页面调整筛选条件，再交给项目轮换关键词。

如果当前页面不是有效搜索页，则从已保存的 Search Plan 生成 BOSS 原生筛选 URL。扫描快照会保存本轮使用的搜索模板，恢复时沿用同一套条件，避免中途漂移。

每轮最多选择 3 个关键词，优先使用当天尚未使用、历史有效率更高的关键词。日常扫描只使用 A/B 关键词；A/B 主档每词最多采集约 50/33 张卡片，右栏详情上限约 45/30。跨关键词和薪资档按岗位 ID 去重。

## 模型调用

首次使用通常包含两类模型调用：

1. `analyzeResume`：把脱敏后的简历文本整理为 CandidateProfile。
2. `recommendSearchPlan`：根据画像推荐岗位方向、关键词、薪资和经验范围；城市由用户选择。

对一个没有命中缓存、且已取得完整 JD 的岗位，语义分析通常包含：

1. `understandJob`：仅理解岗位角色、职责、核心栈和资格条件。
2. `matchJob`：结合候选人画像和具体简历版本判断匹配度并提供双向证据。

本地岗位分析默认并发 3 路，但 BOSS 页面操作始终串行。结构化输出不合约时会追加一次契约修复；仍失败的岗位进入待语义分析，不伪装成推荐结果。模型、契约、画像、简历版本或 JD 变化时缓存会失效。

定制招呼语、HR 回复和无回复跟进只在用户主动点击后调用 `draftCommunication`，不在海投扫描阶段批量生成。

## 清单与批量沟通

- 系统默认勾选可沟通岗位，用户可以逐项取消。
- 清单保存岗位 ID、公司、标题和链接快照，执行时不重新接受一份已变化的列表。
- 用户确认清单后，还要点击“开始沟通”才会访问 BOSS。
- 每个岗位先核验岗位身份和“立即沟通”状态，再执行一次点击；点击后再次核验聊天或按钮状态，无法确认时记录为歧义而不是成功。
- 同一岗位已投、已沟通、失效或存在歧义记录时，不会再次自动进入执行队列。
- 单项之间随机等待 15–20 秒；10 分钟最多 30 个、30 分钟最多 60 个、24 小时最多 150 个。
- 达到本轮成功目标后，未访问替补以零点击停止，继续保留为后续候选。

批量沟通使用 BOSS 平台已有的通用招呼语。模型生成的定制招呼语只供用户查看和手动使用，当前执行器不会自动把生成文本填写或发送给 HR。

## 中断与恢复

- 扫描批次按关键词目标保存检查点；中断后恢复原轮次，不额外占用新轮次。
- `review_required` 阶段可以关闭程序，稍后继续确认清单。
- 沟通执行支持暂停和恢复；遇到未完成的点击状态会要求人工复核，不盲目重放。
- 登录失效、风控、浏览器断开、页面结构变化或岗位身份不一致会立即中断。
- 重启只校准本地状态，不会自动导航、刷新、点击或继续沟通。

## 日常建议

通常一天执行 1–2 轮即可。每轮完成后先看推荐数量与沟通结果；只有候选明显不足且已经满足两小时间隔时，再考虑条件式第三轮。不要为了达到固定数量连续重跑扫描。
